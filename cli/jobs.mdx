## Commands

```sh
operator job list
operator job info <job-name>
operator job run <job-name>
operator job enable <job-name>
operator job disable <job-name>
```

The CLI is for inspection and control. Creating, updating, and deleting jobs is done by editing `JOB.md` files directly or via the runtime tools `create_job`, `update_job`, and `delete_job`.

## `list`

```sh
operator job list
```

Shows one row per job with:

- enabled or disabled state
- schedule
- last run
- last result
- run, error, gate, and skip counters

## `info`

```sh
operator job info daily-summary
```

Shows:

- frontmatter fields like name, schedule, enabled state, description, and path
- runtime state like `last_run`, `last_result`, duration, errors, and counters

## `run`

```sh
operator job run daily-summary
```

This command:

- ignores the cron schedule
- runs immediately
- works even if the job is disabled
- uses a CLI transport instead of Slack

That last point matters: any `send_message` call made by the job is printed to your terminal rather than posted to Slack.

## `enable` / `disable`

```sh
operator job enable daily-summary
operator job disable daily-summary
```

These commands rewrite the `enabled:` field in the job's YAML frontmatter.

You may target:

- the job directory name, when it matches
- or the frontmatter `name`, if different

## Typical Workflow

```sh
operator job list
operator job info daily-summary
operator job run daily-summary
operator job disable daily-summary
```
