---
title: "Service"
description: "Install and manage Operator as a background service"
---

Operator supports:

- `launchd` on macOS
- `systemd --user` on Linux

## Commands

```sh
operator service install
operator service uninstall
operator service start
operator service stop
operator service restart
operator service status
```

## `install`

```sh
operator service install
```

Behavior:

- captures your current shell `PATH`
- writes the service definition
- ensures logs go to `~/.operator/logs/operator.log`

Platform details:

- macOS: unloads the existing service if needed, writes `~/Library/LaunchAgents/ai.operator.plist`, then bootstraps it
- Linux: writes `~/.config/systemd/user/operator.service`, reloads the user daemon, and enables the unit

On Linux, `install` enables the unit but does not start it. Run `operator service start` after installation.

## `uninstall`

```sh
operator service uninstall
```

Platform details:

- macOS: bootouts and removes the plist if present
- Linux: disables and stops the unit, removes the unit file, and reloads the user daemon

## `start`, `stop`, `restart`, `status`

```sh
operator service start
operator service stop
operator service restart
operator service status
```

These proxy to:

- `launchctl` on macOS
- `systemctl --user` on Linux

`status` exits non-zero when the service is not running or not loaded.

## Installed Files

| Platform | Path |
| --- | --- |
| macOS | `~/Library/LaunchAgents/ai.operator.plist` |
| Linux | `~/.config/systemd/user/operator.service` |

## Why PATH Capture Matters

The service definition embeds your current `PATH` so agents can find tools installed via:

- Homebrew
- pyenv
- nvm
- local package managers

If you install new tools later, re-run:

```sh
operator service install
```

## Logs

Both stdout and stderr go to:

```text
~/.operator/logs/operator.log
```

Use:

```sh
operator logs -f
```

to watch the service in real time.
