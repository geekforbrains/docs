The `operator` command is both the runtime entrypoint and the management CLI.

## The Main Rule

Running:

```sh
operator
```

starts the long-running runtime.

That runtime expects:

- a valid `~/.operator/operator.yaml`
- at least one configured transport somewhere under `agents:`

Everything else in the CLI is for setup, inspection, and maintenance.

## Top-Level Commands

| Command | Purpose |
| --- | --- |
| `operator` | start the runtime |
| `operator init` | scaffold `~/.operator/` |
| `operator config` | print resolved config as JSON |
| `operator agents` | list configured agents |
| `operator tools` | list built-in tools |
| `operator logs` | tail `~/.operator/logs/operator.log` |

## Command Groups

| Group | Purpose |
| --- | --- |
| `operator user ...` | users, identities, and roles |
| `operator service ...` | `launchd` or `systemd --user` service management |
| `operator job ...` | inspect, enable, disable, and manually run jobs |
| `operator skills ...` | list discoverable skills and env status |
| `operator memory ...` | inspect memory and rebuild the search index |

Useful defaults:

- `operator memory` is the same as `operator memory list global`
- `operator skills` is the same as `operator skills list`

## Common Flows

### First Install

```sh
pip install operator-ai
operator init
operator user add gavin --role admin slack U123ABC45
operator
```

### Inspect A Running Setup

```sh
operator config
operator agents
operator tools
operator logs -n 100
```

### Run A Job Manually

```sh
operator job list
operator job info daily-summary
operator job run daily-summary
```

### Reindex Memory After Manual Edits

```sh
operator memory index
```
