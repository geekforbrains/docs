The `operator` command is both the long-running runtime entrypoint and the management CLI.

## Scan First

- Running `operator` with no subcommand starts the runtime.
- Most CLI commands work without the background service running.
- `operator tools` shows built-in tools only. Transport tools are runtime-only.

## Top-Level Commands

| Command | Purpose |
| --- | --- |
| `operator` | start the runtime |
| `operator init` | scaffold `~/.operator/` |
| `operator config` | print resolved config as JSON |
| `operator agents` | list configured agents |
| `operator tools` | list built-in tools |
| `operator logs` | tail `operator.log` |

## Command Groups

| Group | Purpose |
| --- | --- |
| `operator user ...` | users, identities, and roles |
| `operator service ...` | launchd/systemd service management |
| `operator job ...` | inspect, enable/disable, and manually run jobs |
| `operator skills ...` | list discoverable skills and env status |
| `operator memory ...` | inspect and reindex file-backed memory |

## Common Flows

### First install

```sh
pip install operator-ai
operator init
operator user add gavin --role admin slack U123ABC45
operator
```

### Inspect a running setup

```sh
operator config
operator agents
operator tools
operator logs -n 100
```

### Service management

```sh
operator service install
operator service start
operator service status
```
