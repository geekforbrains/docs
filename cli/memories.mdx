title: "Memories CLI"

---

Use the memory CLI to inspect what is currently active in the store and to debug why a memory is or is not being recalled.

## Commands

```sh
operator memories [--scope/-s SCOPE] [--scope-id/-i ID] [--pinned] [--limit/-n N]
operator memories stats
```

## List memories

```sh
operator memories
operator memories --scope agent --scope-id researcher
operator memories --scope global --scope-id global --pinned
```

Options:

| Option | Meaning |
| --- | --- |
| `--scope`, `-s` | Filter to `user`, `agent`, or `global` |
| `--scope-id`, `-i` | Filter to a specific scope ID, such as `researcher` or `global` |
| `--pinned` | Show only pinned memories |
| `--limit`, `-n` | Maximum rows to display |

## Debugging workflow

Use this sequence when memory recall feels wrong:

1. Run `operator memories` to confirm the memory exists and is still active.
2. Run `operator memories stats` to see whether you are dealing with mostly `candidate` or `durable` memories.
3. Tail logs to inspect retrieval and injection behavior:

```sh
operator logs -f | rg 'memory recall:|memory injection:'
```

Example log output:

```text
INFO  [researcher:abc123] memory recall: requested_top_k=3 raw_matches=4 filtered_matches=2 returned=2 min_relevance=0.300
INFO  [researcher:abc123] memory injection: injected=2 retention_mix=candidate=1 durable=1
```

Interpretation:

- `raw_matches` is what vector search returned before relevance filtering
- `filtered_matches` survived `memory.inject_min_relevance`
- `returned` is the post-filter count actually handed back to the runtime
- `injected` is what was added to the request context
- `retention_mix` shows whether recall is being dominated by short-lived or durable memory

If `raw_matches` is high but `filtered_matches` is low, your relevance threshold is probably aggressive. If `returned` is lower than expected, reduce `inject_min_relevance` or increase `inject_top_k`.
