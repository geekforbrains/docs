This page covers the top-level commands outside the grouped subcommands.

## Start The Runtime

```sh
operator
```

What it does:

- loads `~/.operator/operator.yaml`
- loads `runtime.env_file` if configured
- ensures the full `~/.operator/` layout exists
- builds the memory search index from files
- starts the configured transports
- starts the job runner
- starts the hourly memory expiry sweep

The runtime requires at least one configured transport. A headless-only config cannot start the long-running service.

## `operator init`

```sh
operator init
operator init --force
```

Scaffolds:

- `operator.yaml`
- `.env`
- `SYSTEM.md`
- `agents/operator/AGENT.md`
- the full workspace, memory, state, jobs, skills, shared, db, and logs directories

Behavior:

- safe to re-run
- existing `.env` and `AGENT.md` files are preserved
- existing `operator.yaml` is replaced only after confirmation, unless `--force` is used

After scaffolding, the CLI tells you to edit `operator.yaml`, add secrets to `.env`, and create your first admin user.

## `operator config`

```sh
operator config
```

Prints the resolved config as JSON after validation and env-file loading.

Use this to confirm:

- agent inheritance
- role definitions
- permission groups
- transport config normalization
- env file loading

## `operator agents`

```sh
operator agents
```

Shows:

- configured agent name from `operator.yaml`
- description from `AGENT.md` when present, otherwise a fallback placeholder
- transport type
- resolved models
- whether `AGENT.md` and `workspace/` exist

This command lists configured agents, not arbitrary folders on disk, so it is the quickest way to confirm that runtime config and on-disk scaffold agree.

## `operator tools`

```sh
operator tools
```

Lists the registered built-in tools.

Important caveat: this command does **not** include transport tools like Slack's `slack_list_channels`, `slack_read_channel`, or `slack_read_thread`. Those are added only during live runtime execution.

## `operator logs`

```sh
operator logs
operator logs -n 200
operator logs -f
```

Tails `~/.operator/logs/operator.log`.

Options:

| Option | Meaning |
| --- | --- |
| `-n`, `--lines` | number of lines to show, default `50` |
| `-f`, `--follow` | follow new output |

If the log file does not exist yet, the command exits with an error.

## Module Entrypoint

If the `operator` script is not on your `PATH`, the package also supports:

```sh
python3 -m operator_ai
python3 -m operator_ai init
```
