Runtime config lives at `~/.operator/operator.yaml`.

## Scan First

- `defaults.models` is required and must contain at least one LiteLLM model string.
- `admin` is a built-in role and cannot be defined under `roles:`.
- Agent permissions are closed by default. If an agent has no `permissions` block, it gets no tools and no skills.
- Slack is the only supported `transport.type` right now.
- Memory is file-backed. Optional embeddings are configured under `defaults.embeddings`.

## Full Example

```yaml
runtime:
  env_file: ".env"
  show_usage: false
  reject_response: ignore

defaults:
  models:
    - "anthropic/claude-sonnet-4-6"
    - "openai/gpt-4.1"
  thinking: medium
  max_iterations: 25
  context_ratio: 0.5
  max_output_tokens: null
  hook_timeout: 30
  embeddings:
    model: "openai/text-embedding-3-small"
    dimensions: 1536

permission_groups:
  files:
    - read_file
    - write_file
    - list_files
  web:
    - web_fetch
  agents:
    - spawn_agent

agents:
  operator:
    permissions:
      tools: "*"
      skills: "*"
    transport:
      type: slack
      env:
        bot_token: SLACK_BOT_TOKEN
        app_token: SLACK_APP_TOKEN
      settings:
        include_archived_channels: false
        inject_channels_into_prompt: true
        inject_users_into_prompt: true
        expand_mentions: true

  researcher:
    models:
      - "openai/o3"
    thinking: high
    permissions:
      tools: [@files, @web, run_shell, read_skill, run_skill, @agents]
      skills: [pr-reviewer]

roles:
  team:
    agents: [operator, researcher]
  guest:
    agents: []
```

## `runtime`

```yaml
runtime:
  env_file: ".env"
  show_usage: false
  reject_response: ignore
```

| Field | Default | What it does |
| --- | --- | --- |
| `env_file` | `null` | Optional `.env` file path. Relative paths resolve from the config file directory |
| `show_usage` | `false` | Sends a usage summary after each completed chat response |
| `reject_response` | `ignore` | Unknown or unauthorized user behavior: `ignore` or `announce` |

<Note>
  There is no `runtime.timezone` setting in current Operator. Timezone is stored on each user record and can be set in chat with `set_timezone("America/Vancouver")`. Live user-message timestamps render in that user timezone when available. Jobs run without a user context and therefore render in UTC.
</Note>

### `runtime.env_file`

- Paths support `~` expansion.
- Relative paths resolve from the config file directory.
- Existing environment variables are not overwritten by the file.

## `defaults`

```yaml
defaults:
  models:
    - "anthropic/claude-sonnet-4-6"
  thinking: off
  max_iterations: 25
  context_ratio: 0.5
  max_output_tokens: null
  hook_timeout: 30
  embeddings: null
```

| Field | Default | What it does |
| --- | --- | --- |
| `models` | required | Fallback chain of LiteLLM model IDs |
| `thinking` | `off` | Reasoning level: `off`, `low`, `medium`, `high` |
| `max_iterations` | `25` | Max model/tool loop iterations |
| `context_ratio` | `0.5` | Fraction of the model input window Operator is willing to spend on live request context before each call. Must be greater than `0` and at most `1`. |
| `max_output_tokens` | `null` | Per-call output cap. `null` means “use the model max when LiteLLM can resolve it” |
| `hook_timeout` | `30` | Timeout in seconds for job prerun/postrun hooks |
| `embeddings` | `null` | Optional embedding model config for semantic note search via `sqlite-vec` |

### `models`

- Uses LiteLLM format: `provider/model-name`
- The first model is preferred
- Later models are fallbacks for live request failures

Operator also accepts a singular `model:` key in `defaults` or per-agent config and normalizes it into `models: [...]`. The docs use list form because that is the real fallback surface.

### `context_ratio`

When a live request exceeds the model budget, Operator:

1. measures the full outbound payload, including messages and tool schemas
2. keeps the system prompt
3. keeps the latest user exchange
4. drops older exchange groups first

There is no special `0` value. `context_ratio` must be greater than `0`.

### `embeddings`

```yaml
defaults:
  embeddings:
    model: "openai/text-embedding-3-small"
    dimensions: 1536
```

When configured, Operator builds vector embeddings for note memories and combines them with the SQLite FTS5 index under `~/.operator/db/memory_index.db`.

| Field | Default | Notes |
| --- | --- | --- |
| `model` | required | LiteLLM embedding model ID |
| `dimensions` | `1536` | Vector size for the sqlite-vec table |

If `sqlite-vec` is not installed, Operator logs a warning and continues with FTS-only note search.

## `permission_groups`

```yaml
permission_groups:
  files:
    - read_file
    - write_file
    - list_files
  web:
    - web_fetch
```

Permission groups are named bundles you can reference as `@group` in an agent's permission lists.

```yaml
agents:
  researcher:
    permissions:
      tools: [@files, @web, run_shell]
```

Unknown groups log a warning and are ignored.

## `agents`

```yaml
agents:
  operator:
    models:
      - "anthropic/claude-sonnet-4-6"
    thinking: medium
    max_iterations: 40
    context_ratio: 0.4
    max_output_tokens: 4096
    transport:
      type: slack
      env:
        bot_token: SLACK_BOT_TOKEN
        app_token: SLACK_APP_TOKEN
    permissions:
      tools: [read_file, list_files, web_fetch]
      skills: [pr-reviewer]
```

The configured key under `agents:` is the runnable agent name and the value used by `spawn_agent(agent="...")`. `AGENT.md` supplies the prompt body and optional human-facing description.

Each agent can override any of the `defaults` execution knobs plus transport and permissions.

| Field | Default | Notes |
| --- | --- | --- |
| `models` | inherit from `defaults.models` | Fallback chain for this agent |
| `thinking` | inherit from `defaults.thinking` | Simple reasoning level |
| `max_iterations` | inherit from `defaults.max_iterations` | Max tool loop depth |
| `context_ratio` | inherit from `defaults.context_ratio` | Live request budget fraction |
| `max_output_tokens` | inherit from `defaults.max_output_tokens` | Output token cap |
| `transport` | none | Makes the agent reachable from Slack |
| `permissions` | none | Optional flat allow-lists for tools and skills |

Agents without a `transport` block are headless: they can still be used for jobs and `spawn_agent(...)`, but they are not directly reachable in chat.

<Warning>
  The long-running `operator` service still requires at least one agent with a transport. A config containing only headless agents will not start the runtime.
</Warning>

### `permissions`

```yaml
permissions:
  tools: [@files, web_fetch, spawn_agent]
  skills: [pr-reviewer]
```

| Value | Meaning |
| --- | --- |
| omitted | deny all |
| `null` | deny all |
| `"*"` | unrestricted |
| list | only the listed names are available |

Permissions are allow-lists. Unlisted tools and skills are removed from the LLM-visible tool and skill surface.

Tool permissions and skill permissions are independent.

## `transport`

```yaml
transport:
  type: slack
  env:
    bot_token: SLACK_BOT_TOKEN
    app_token: SLACK_APP_TOKEN
  settings:
    include_archived_channels: false
    inject_channels_into_prompt: true
    inject_users_into_prompt: true
    expand_mentions: true
```

`transport.type` must currently be `slack`.

| Field | Required | Notes |
| --- | --- | --- |
| `type` | yes | Only `slack` is supported |
| `env.bot_token` | yes | Env var name for the bot token |
| `env.app_token` | yes | Env var name for the Socket Mode app token |
| `settings.include_archived_channels` | no | Include archived channels in the cached channel list |
| `settings.inject_channels_into_prompt` | no | Inject the cached channel list into the system prompt |
| `settings.inject_users_into_prompt` | no | Inject the Slack user list into the system prompt |
| `settings.expand_mentions` | no | Expand Slack mentions and channel references into readable text in inbound messages |

Tokens are resolved from the process environment after `runtime.env_file` is loaded.

## `roles`

```yaml
roles:
  team:
    agents: [operator, researcher]
  guest:
    agents: []
```

Roles define which agents a user may message.

| Rule | Behavior |
| --- | --- |
| `admin` | built-in, grants access to all agents, cannot be redefined |
| multiple roles | allowed agents are the union of all listed agent names |
| undefined custom role | grants no agent access |

`operator user add` warns when you create a non-admin user with a role name that is not defined in config.

## Memory And State

Current Operator has no top-level `memory:` or `state:` config sections.

- Memory files live under `~/.operator/memory/` and `~/.operator/agents/<agent>/memory/`.
- The search index lives at `~/.operator/db/memory_index.db`.
- Optional semantic search is configured with `defaults.embeddings`.
- Agent state is stored as JSON files under `~/.operator/agents/<agent>/state/`.

See [Memory](/concepts/memory) and [Tools](/concepts/tools) for the runtime behavior of rules, notes, and state files.
