Runtime config lives at `~/.operator/operator.yaml`.

Start with the generated file from `operator init`. Most teams only need to change four things at first:

1. `defaults.models`
2. the body of `AGENT.md`
3. `roles`
4. agent `permissions`

Everything else is there when you need it.

## Mental Model

`operator.yaml` has five top-level sections:

- `runtime`: process-level behavior such as `.env` loading and rejection policy
- `defaults`: model and execution defaults inherited by agents
- `permission_groups`: named bundles you can reference as `@group`
- `agents`: the runnable agents
- `roles`: which humans can talk to which agents

Unknown keys are rejected. The config models are strict, so typos fail fast instead of silently doing nothing.

## Full Example

```yaml
runtime:
  env_file: ".env"
  show_usage: false
  reject_response: ignore

defaults:
  models:
    - "anthropic/claude-sonnet-4-6"
    - "openai/gpt-4.1"
  thinking: medium
  max_iterations: 25
  context_ratio: 0.5
  max_output_tokens: null
  hook_timeout: 30
  embeddings:
    model: "openai/text-embedding-3-small"
    dimensions: 1536

permission_groups:
  files:
    - read_file
    - write_file
    - list_files
  web:
    - web_fetch
  agents:
    - spawn_agent

agents:
  operator:
    permissions:
      tools: "*"
      skills: "*"
    transport:
      type: slack
      env:
        bot_token: SLACK_BOT_TOKEN
        app_token: SLACK_APP_TOKEN
      settings:
        include_archived_channels: false
        inject_channels_into_prompt: true
        inject_users_into_prompt: true
        expand_mentions: true

  researcher:
    models:
      - "openai/o3"
    thinking: high
    permissions:
      tools: [@files, @web, read_skill, run_skill, @agents]
      skills: [docs-auditor]

roles:
  team:
    agents: [operator, researcher]
  guest:
    agents: []
```

## `runtime`

```yaml
runtime:
  env_file: ".env"
  show_usage: false
  reject_response: ignore
```

| Field | Default | Meaning |
| --- | --- | --- |
| `env_file` | `null` | optional dotenv-style file path |
| `show_usage` | `false` | posts token usage after successful chat replies |
| `reject_response` | `ignore` | what to do when an unknown or unauthorized user messages an agent: `ignore` or `announce` |

Technical details:

- relative `env_file` paths resolve from the config file directory
- `~` expansion is supported
- loaded values do not overwrite environment variables already set in the shell

There is no runtime-wide timezone setting. User timezone is stored per user record and can be set during chat with `set_timezone("America/Vancouver")`.

## `defaults`

```yaml
defaults:
  models:
    - "anthropic/claude-sonnet-4-6"
  thinking: off
  max_iterations: 25
  context_ratio: 0.5
  max_output_tokens: null
  hook_timeout: 30
  embeddings: null
```

| Field | Default | Meaning |
| --- | --- | --- |
| `models` | required | fallback chain of LiteLLM model IDs |
| `thinking` | `off` | reasoning level: `off`, `low`, `medium`, `high` |
| `max_iterations` | `25` | max LLM/tool loop iterations |
| `context_ratio` | `0.5` | fraction of the model input window Operator will spend on live request context |
| `max_output_tokens` | `null` | explicit per-call output cap; otherwise Operator asks LiteLLM for model max output when available |
| `hook_timeout` | `30` | timeout in seconds for job prerun and postrun hooks |
| `embeddings` | `null` | optional semantic note-search config |

Important behavior:

- `models` is a fallback chain, not just a preferred model
- the singular alias `model:` is accepted and normalized to `models: [...]`
- `context_ratio` must be greater than `0` and at most `1`
- when `thinking` is unsupported by the concrete model, Operator drops the provider-specific control and continues

### `embeddings`

```yaml
defaults:
  embeddings:
    model: "openai/text-embedding-3-small"
    dimensions: 1536
```

This enables vector embeddings for note search alongside SQLite FTS5.

Requirements:

- configure an embedding model
- install the optional dependency:

```sh
pip install sqlite-vec
```

If `sqlite-vec` is missing, Operator logs a warning and continues with FTS-only search.

## `permission_groups`

```yaml
permission_groups:
  files:
    - read_file
    - write_file
    - list_files
  web:
    - web_fetch
```

Permission groups are named bundles referenced as `@group` inside an agent's permission lists:

```yaml
agents:
  researcher:
    permissions:
      tools: [@files, @web, run_shell]
```

This keeps agent permissions readable and makes it easier to share the same tool bundles across multiple agents.

## `agents`

Each key under `agents:` is a runnable identity. That key is the source of truth for the runtime name.

```yaml
agents:
  operator:
    models:
      - "anthropic/claude-sonnet-4-6"
    thinking: off
    max_iterations: 25
    context_ratio: 0.5
    max_output_tokens: null
    permissions:
      tools: "*"
      skills: "*"
    transport:
      type: slack
      env:
        bot_token: SLACK_BOT_TOKEN
        app_token: SLACK_APP_TOKEN
      settings:
        include_archived_channels: false
        inject_channels_into_prompt: true
        inject_users_into_prompt: true
        expand_mentions: true
```

Per-agent overrides are optional. If a field is omitted, Operator falls back to `defaults`.

### Permissions

Permissions are closed by default.

- no `permissions` block means no tools and no skills
- `tools: "*"` or `skills: "*"` means full access
- lists are allowlists

Example:

```yaml
permissions:
  tools: [@files, read_skill, run_skill, spawn_agent]
  skills: [docs-auditor]
```

Tool permissions and skill permissions are separate. An agent may be allowed to discover a skill but still lack a tool such as `run_shell`.

### Transport

Current code supports only one runtime transport type:

```yaml
transport:
  type: slack
```

Slack transport config has two parts:

- `env`: names of environment variables that hold secrets
- `settings`: non-secret behavior flags

```yaml
transport:
  type: slack
  env:
    bot_token: SLACK_BOT_TOKEN
    app_token: SLACK_APP_TOKEN
  settings:
    include_archived_channels: false
    inject_channels_into_prompt: true
    inject_users_into_prompt: true
    expand_mentions: true
```

Technical details:

- config normalization lowercases the transport type
- missing or extra Slack config keys fail validation
- agents without a transport are still valid for jobs and `spawn_agent`
- the runtime process still needs at least one transport somewhere in `agents:` or startup exits

## `roles`

Roles decide which humans can talk to which agents.

```yaml
roles:
  team:
    agents: [operator, researcher]
```

Rules:

- `admin` is built in and cannot be defined in config
- access is the union of all of a user's roles
- undefined roles can still be stored on users, but they grant no access until they appear in config

## Inheritance Rules

Use this as the quick reference:

| Setting | Chat run | Job run |
| --- | --- | --- |
| `models` | agent override or `defaults.models` | `job.model` or resolved agent models |
| `thinking` | agent override or `defaults.thinking` | resolved agent thinking |
| `max_iterations` | agent override or `defaults.max_iterations` | `job.max_iterations` or resolved agent value |
| `context_ratio` | agent override or `defaults.context_ratio` | resolved agent value |
| `max_output_tokens` | agent override or `defaults.max_output_tokens` | resolved agent value |

If you need a scheduled task to use a different `thinking`, `context_ratio`, or permission set, point that job at a different agent rather than trying to override the job itself.

## Practical Example

This is a common pattern:

- one Slack-connected `operator` agent for humans
- one headless `researcher` agent for deep work
- humans can talk to both or only to `operator`, depending on role
- `operator` delegates focused tasks with `spawn_agent(agent="researcher", ...)`

That keeps the chat-facing agent broad and safe while giving the specialist agent a narrower tool set and stronger reasoning model.
