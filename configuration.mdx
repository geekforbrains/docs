Runtime config lives at `~/.operator/operator.yaml`.

Start with the file generated by `operator init`. Most teams only need to change four things at first:

1. `defaults.models`
2. the body of `AGENT.md`
3. `roles`
4. each agent's `permissions`

Everything else is there when you need it.

## Mental Model

`operator.yaml` has five top-level sections:

- `runtime`: process-level behavior like `.env` loading and rejection policy
- `defaults`: model and execution defaults inherited by agents
- `permission_groups`: named bundles you can reference as `@group`
- `agents`: the runnable agents
- `roles`: which users can talk to which agents

Config models are strict. Unknown keys fail fast instead of silently doing nothing.

## Full Example

```yaml
runtime:
  env_file: ".env"
  show_usage: false
  reject_response: ignore

defaults:
  models:
    - "anthropic/claude-sonnet-4-6"
    - "openai/gpt-4.1"
  thinking: medium
  max_iterations: 25
  context_ratio: 0.5
  max_output_tokens: null
  hook_timeout: 30
  embeddings:
    model: "openai/text-embedding-3-small"
    dimensions: 1536

permission_groups:
  files:
    - read_file
    - write_file
    - list_files
  web:
    - web_fetch
  agents:
    - spawn_agent

agents:
  operator:
    permissions:
      tools: "*"
      skills: "*"
    transport:
      type: slack
      env:
        bot_token: SLACK_BOT_TOKEN
        app_token: SLACK_APP_TOKEN
      settings:
        include_archived_channels: false
        inject_channels_into_prompt: true
        inject_users_into_prompt: true
        expand_mentions: true

  researcher:
    models:
      - "openai/o3"
    thinking: high
    permissions:
      tools: [@files, @web, read_skill, run_skill, @agents]
      skills: [docs-auditor]

roles:
  team:
    agents: [operator, researcher]
  guest:
    agents: []
```

## `runtime`

```yaml
runtime:
  env_file: ".env"
  show_usage: false
  reject_response: ignore
```

| Field | Default | Meaning |
| --- | --- | --- |
| `env_file` | `null` | optional dotenv-style file path |
| `show_usage` | `false` | post token usage after successful chat replies |
| `reject_response` | `ignore` | what to do when an unknown or unauthorized user messages an agent: `ignore` or `announce` |

Technical details:

- relative `env_file` paths resolve from the config directory
- `~` expansion is supported
- loaded values do not override environment variables already set in the shell

User timezone is not configured here. It is stored per user record and exposed in the runtime context for live conversations.

## `defaults`

```yaml
defaults:
  models:
    - "anthropic/claude-sonnet-4-6"
  thinking: off
  max_iterations: 25
  context_ratio: 0.5
  max_output_tokens: null
  hook_timeout: 30
  embeddings: null
```

| Field | Default | Meaning |
| --- | --- | --- |
| `models` | required | fallback chain of LiteLLM model IDs |
| `thinking` | `off` | reasoning level: `off`, `low`, `medium`, `high` |
| `max_iterations` | `25` | max LLM and tool loop iterations |
| `context_ratio` | `0.5` | fraction of the model input budget used for live request context |
| `max_output_tokens` | `null` | explicit per-call output cap |
| `hook_timeout` | `30` | timeout in seconds for job hooks |
| `embeddings` | `null` | optional semantic note-search config |

Important behavior:

- `models` is a fallback chain, not just a preference list
- `model:` is accepted as a singular alias and normalized to `models: [...]`
- `context_ratio` must be greater than `0` and at most `1`
- when a provider does not support reasoning control, Operator drops the provider-specific setting and continues

### `embeddings`

```yaml
defaults:
  embeddings:
    model: "openai/text-embedding-3-small"
    dimensions: 1536
```

This enables vector search alongside SQLite FTS5 for notes.

Requirements:

1. configure an embedding model
2. install the optional dependency

```sh
pip install sqlite-vec
```

If `sqlite-vec` is missing, Operator logs a warning and continues with FTS-only search.

## `permission_groups`

Permission groups keep agent permissions readable.

```yaml
permission_groups:
  files:
    - read_file
    - write_file
    - list_files
  web:
    - web_fetch

agents:
  researcher:
    permissions:
      tools: [@files, @web, run_shell]
```

Unknown `@group` names are warned about during permission expansion and do not grant access.

## `agents`

Each key under `agents:` is a runnable identity. That key is the source of truth for the runtime name.

```yaml
agents:
  operator:
    models:
      - "anthropic/claude-sonnet-4-6"
    thinking: off
    max_iterations: 25
    context_ratio: 0.5
    max_output_tokens: null
    permissions:
      tools: "*"
      skills: "*"
    transport:
      type: slack
      env:
        bot_token: SLACK_BOT_TOKEN
        app_token: SLACK_APP_TOKEN
      settings:
        include_archived_channels: false
        inject_channels_into_prompt: true
        inject_users_into_prompt: true
        expand_mentions: true
```

Per-agent overrides are optional. If a field is omitted, Operator falls back to `defaults`.

### Permissions

Permissions are closed by default.

- no `permissions` block means no tools and no skills
- `tools: "*"` means all built-in and runtime-added tools
- `skills: "*"` means all discovered skills
- lists restrict access to named tools or named skills

Example:

```yaml
permissions:
  tools: [read_file, write_file, list_files, web_fetch, spawn_agent]
  skills: [docs-auditor]
```

### Transport

Transport config has three parts:

- `type`
- `env`
- `settings`

For Slack, the required fields are:

- `type: slack`
- `env.bot_token`
- `env.app_token`

`env` maps logical credential names to environment variable names. `settings` controls non-secret runtime behavior.

Current Slack `settings` keys:

| Key | Default | What it does |
| --- | --- | --- |
| `include_archived_channels` | `false` | include archived channels in the cached Slack channel snapshot |
| `inject_channels_into_prompt` | `true` | inject the cached channel list into the system prompt |
| `inject_users_into_prompt` | `true` | inject the cached workspace member list into the system prompt |
| `expand_mentions` | `true` | rewrite outbound `@Name` and `#channel` text to explicit Slack mention tokens before sending |

Important distinction:

- `inject_channels_into_prompt` and `inject_users_into_prompt` change prompt content
- `expand_mentions` changes outbound message rendering only

## `roles`

Roles map users to allowed agents.

```yaml
roles:
  team:
    agents: [operator, researcher]
```

Rules:

- `admin` is built in and cannot be redefined
- a user's access is the union of all of their roles
- roles not defined in config can still exist in the database, but they grant no access until you define them here

## Inheritance Rules

| Setting | Resolution |
| --- | --- |
| `models` | agent override or `defaults.models` |
| `thinking` | agent override or `defaults.thinking` |
| `max_iterations` | agent override or `defaults.max_iterations` |
| `context_ratio` | agent override or `defaults.context_ratio` |
| `max_output_tokens` | agent override or `defaults.max_output_tokens` |

Jobs may override `model` and `max_iterations` in their own `JOB.md` frontmatter. They do not override `thinking`, `context_ratio`, or `max_output_tokens`.

## Practical Patterns

Start simple:

- one agent with Slack transport
- one admin user
- `tools: "*"` and `skills: "*"` until you know what to tighten

Then harden:

- split out specialist agents
- replace `*` with named tool lists or `@group`s
- give `manage_users`, `run_shell`, and `write_file` only to trusted agents
- keep public-facing agents narrow and boring
