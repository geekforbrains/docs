Runtime config lives at `~/.operator/operator.yaml`.

## Runtime

```yaml
runtime:
  timezone: "America/Vancouver"
  env_file: ".env"
  show_usage: false
  reject_response: ignore
```

| Field | Description |
|-------|-------------|
| `timezone` | IANA timezone used for job scheduling, Slack message formatting, the system prompt's default timezone assumption, and request-time current-time injection. Defaults to `UTC`. |
| `env_file` | Path to env file for API keys. Relative paths resolve from `~/.operator/`. Supports `~` expansion. |
| `show_usage` | Append token usage stats as a reply after each agent response. Defaults to `false`. |
| `reject_response` | What happens when an unauthorized user messages an agent. `announce` replies with "You don't have access to this agent." `ignore` silently drops the message. Defaults to `ignore`. |

Operator uses `runtime.timezone` in two places:

- The stable runtime block tells agents to interpret ambiguous dates and times in that timezone unless the user says otherwise.
- Right before each live model call, Operator injects a fresh current-time block into the request payload for chats, jobs, and sub-agents.

That current-time block is request-only. It is not written to persisted conversation history, does not become part of the stable system prompt, and does not disturb the cacheable prompt prefix.

## Defaults

```yaml
defaults:
  models:
    - "anthropic/claude-opus-4-6"
    - "openai/gpt-4.1"
  thinking: off
  max_iterations: 25
  context_ratio: 0.5
  max_output_tokens: null
```

| Field | Description |
|-------|-------------|
| `models` | Fallback chain â€” if the first model errors, the next is tried. Always use list format. |
| `thinking` | Reasoning level for supported models: `off`, `low`, `medium`, or `high`. Defaults to `off`. |
| `max_iterations` | Maximum tool-call loops per agent run. |
| `context_ratio` | Fraction of context window to use for conversation history. |
| `max_output_tokens` | Cap output length per LLM call. `null` uses each model's max. |

### Thinking

Operator keeps reasoning control deliberately simple:

```yaml
defaults:
  thinking: off

agents:
  researcher:
    thinking: high
```

- `off`
- `low`
- `medium`
- `high`

Operator maps these to LiteLLM `reasoning_effort` when the concrete model supports it. If a model in the fallback chain does not support reasoning control, Operator omits the param and continues normally.

This capability check happens per model call, so fallback chains can safely move between reasoning-capable and non-capable models.

<Tip>See [Models & Thinking](/concepts/models) for inheritance, fallback, and job behavior.</Tip>

## Agents

```yaml
agents:
  operator:
    models:
      - "anthropic/claude-sonnet-4-6"
    thinking: medium
    transport:
      type: slack
      bot_token_env: SLACK_BOT_TOKEN
      app_token_env: SLACK_APP_TOKEN
```

Each agent can override `models`, `thinking`, `max_iterations`, `context_ratio`, and `max_output_tokens` from defaults. Agents without a `transport` block are available for jobs and sub-agent spawning but have no chat interface.

### Sandbox

```yaml
agents:
  operator:
    sandbox: false
```

| Field | Default | Description |
|-------|---------|-------------|
| `sandbox` | `true` | When `true`, file tools (`read_file`, `write_file`, `list_files`, `send_file`) are confined to the agent's workspace. When `false`, file tools accept absolute paths and paths outside the workspace. |

`run_shell` is not affected â€” use `permissions.tools` to restrict shell access.

Sub-agents inherit their parent's sandbox setting unless spawned as a different agent (via `spawn_agent(agent="...")`), in which case the target agent's sandbox setting is used.

See [Agents â€” Sandbox](/concepts/agents#sandbox) for usage guidance.

### Permissions

Restrict which tools or skills an agent can use:

```yaml
agents:
  restricted-agent:
    permissions:
      tools: [read_file, list_files, web_fetch, read_skill, run_skill, search_memories]
      skills: [summarize, translate]
```

- No `permissions` block = full access (default).
- `"*"` = explicit full access.
- `[list]` = only these names are available; everything else is hidden.
- Permissions use flat allow-lists.

<Tip>Run `operator tools` to see all available built-in tool names.</Tip>

See [Agents](/concepts/agents) for details on agent prompts and workspaces.

### Shared Directory

All agents share a `~/.operator/shared/` directory, symlinked into each workspace as `workspace/shared/`. Use this for files that need to be accessible across agents â€” shared data, output handoffs, etc. The directory is created by `operator init` and the symlink is set up automatically on each agent run.

## Roles

```yaml
roles:
  team:
    agents: [operator, researcher]
  viewer:
    agents: [researcher]
```

Roles control which agents a user can message. Manage users via the CLI (`operator user`) or the `manage_users` tool. See the [Users](/concepts/users) guide.

- `admin` is a built-in role that grants access to **all** agents. It cannot be redefined in config.
- A user's allowed agents are the **union** of all their role agent lists.

See [Users](/concepts/users) for full details on user management and role assignment.

## Memory

```yaml
memory:
  embed_model: "openai/text-embedding-3-small"
  embed_dimensions: 1536
  max_memories: 10000
  inject_top_k: 3
  inject_min_relevance: 0.3
  candidate_ttl_days: 14
  harvester:
    enabled: true
    schedule: "*/30 * * * *"
    model: "openai/gpt-4.1-mini"
  cleaner:
    enabled: true
    schedule: "0 3 * * *"
    model: "anthropic/claude-haiku-4-5"
```

| Field | Description |
|-------|-------------|
| `embed_model` | Required when any memory service is enabled. |
| `embed_dimensions` | Embedding vector size. |
| `max_memories` | Per-scope soft cap. |
| `inject_top_k` | Maximum memories injected per message after filtering. Set `0` to disable automatic semantic recall. |
| `inject_min_relevance` | Minimum cosine similarity required for a memory to survive retrieval filtering. |
| `candidate_ttl_days` | Age-based TTL for `candidate` memories. `durable` memories do not expire automatically. |

### Harvester

Extracts reusable facts from conversations using an LLM and stores them as vector embeddings in SQLite (via sqlite-vec). The harvester classifies memories as `candidate` or `durable` and is intentionally strict about rejecting thread-local noise. When enabled, `schedule` and `model` are required.

### Cleaner

Deduplicates, merges, and tidies active memories by sending them through an LLM. Expired candidate memories are excluded before the cleaner runs. When enabled, `schedule` and `model` are required.

### Recall debugging

Operator emits retrieval and injection logs so you can tune memory without code changes. Expect lines like:

```text
INFO  [researcher:abc123] memory recall: requested_top_k=3 raw_matches=4 filtered_matches=2 returned=2 min_relevance=0.300
INFO  [researcher:abc123] memory injection: injected=2 retention_mix=candidate=1 durable=1
```

Use `inject_top_k`, `inject_min_relevance`, and `candidate_ttl_days` together:

- Lower `inject_top_k` if too much context is being injected
- Raise `inject_min_relevance` if recall is noisy
- Raise `candidate_ttl_days` if short-term context disappears too quickly

See [Memory](/concepts/memory) for more on scopes, retention, pinning, and retrieval.

## Jobs and Inheritance

Jobs inherit the resolved agent config for:

- `thinking`
- `context_ratio`
- `max_output_tokens`
- permissions
- sandboxing

Job frontmatter can override:

- `model`
- `max_iterations`

There is currently no per-job `thinking` field. If you need a different reasoning level for a job, create or target a different agent config.

## Environment Variables

API keys are resolved from the environment, loaded via `runtime.env_file`. In the starter config generated by `operator init`, this is set to `.env` (relative to `~/.operator/`):

```sh
# LLM providers
ANTHROPIC_API_KEY=sk-ant-...
OPENAI_API_KEY=sk-...

# Slack (per agent)
SLACK_BOT_TOKEN=xoxb-...
SLACK_APP_TOKEN=xapp-...
```

All model names use [LiteLLM format](https://docs.litellm.ai/docs/providers) (`provider/model-name`).

### Runtime Variables

Operator sets these environment variables automatically. They are available in `run_shell` commands, skill scripts, and job hooks:

| Variable | Description |
|----------|-------------|
| `OPERATOR_HOME` | Resolved absolute path to the Operator base directory (typically `~/.operator`). Use this instead of `~/.operator` in scripts and skill instructions â€” tilde expansion is unreliable outside shell contexts. |
| `OPERATOR_AGENT` | Name of the agent running the current job (set in job hooks). |
| `OPERATOR_DB` | Path to the SQLite database (set in job hooks). |
| `JOB_NAME` | Name of the current job (set in job hooks). |
