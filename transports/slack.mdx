Operator uses Slack Socket Mode. You do not need a public webhook URL.

Slack is the only runtime transport in the current code.

## What Slack Means In Operator

- DMs and channel mentions are the inbound entry points
- sessions are thread-scoped on purpose
- Slack adds runtime-only tools for reading channels, threads, users, and reactions
- user and channel snapshots can be injected into the system prompt

## 1. Create The App

Create a Slack app and enable **Socket Mode**.

Create an app-level token with:

- `connections:write`

That becomes `SLACK_APP_TOKEN`.

If you want users to DM the bot, also open **App Home** and enable the **Messages** tab. Operator does not publish a custom Home tab, but it does listen for direct messages via Slack's app-user conversation surface.

## 2. Add Bot Scopes

Under **OAuth & Permissions**, add these bot scopes:

- `app_mentions:read`
- `channels:history`
- `channels:read`
- `chat:write`
- `files:read`
- `files:write`
- `groups:history`
- `groups:read`
- `im:history`
- `im:read`
- `im:write`
- `reactions:read`
- `reactions:write`
- `users:read`

Optional but useful:

- `users:read.email` if you want Slack email addresses available in `slack_find_users`

Why these exist in the current implementation:

- message ingress needs DM and mention access
- Slack read tools need channel and thread history
- outbound messaging and uploads need write scopes
- reaction tools and reaction system events need reaction access
- prompt injection and name resolution need user and channel reads

## 3. Subscribe To Events

Subscribe to these bot events for the default runtime behavior:

- `app_mention`
- `message.im`
- `reaction_added`
- `reaction_removed`

To match the transport's cache-refresh behavior more closely, also subscribe to:

- `team_join`
- `user_change`
- `channel_created`
- `channel_rename`
- `channel_archive`
- `channel_unarchive`

Those extra events are not required for basic messaging, but the code does listen for them to keep its in-memory user and channel snapshots fresher between restarts.

## 4. Configure Operator

Put the Slack secrets in `~/.operator/.env`:

```sh
SLACK_BOT_TOKEN=xoxb-...
SLACK_APP_TOKEN=xapp-...
```

Configure the agent:

```yaml
agents:
  operator:
    transport:
      type: slack
      env:
        bot_token: SLACK_BOT_TOKEN
        app_token: SLACK_APP_TOKEN
      settings:
        include_archived_channels: false
        inject_channels_into_prompt: true
        inject_users_into_prompt: true
        expand_mentions: true
```

## Session Model

Slack conversations are intentionally thread-scoped.

- a top-level DM starts a new session
- a top-level channel mention starts a new thread session
- later replies count as new input only when they mention the agent again
- unmentioned chatter in the thread is ambient context unless the agent reads it deliberately with Slack tools

Conversation IDs are built like this:

```text
slack:<agent-name>:<channel-id>:<root-thread-ts>
```

That keeps each Slack thread isolated as a conversation.

## What Slack Injects

Two different Slack context paths feed the model.

System prompt extras:

- thread-scoping instructions
- the current workspace user list when `inject_users_into_prompt: true`
- the current visible channel list when `inject_channels_into_prompt: true`

Per-message context snapshots:

- current message metadata such as `message_id`, `channel_id`, and `thread_id`
- a one-time snapshot of earlier thread replies when the agent is first brought into an existing thread
- buffered reaction events from messages already tied to the current conversation

`expand_mentions` is separate from prompt injection. It controls outbound message rendering, rewriting plain `@Name` or `#channel` text into explicit Slack mention tokens before `send_message` or message updates are posted.

The channel and user snapshots are loaded on startup and refreshed again when the subscribed lifecycle events arrive.

## Slack Runtime Tools

These tools exist only during live Slack-backed runs:

| Tool | Use it for |
| --- | --- |
| `slack_find_users(query="", limit=20, linked_only=false)` | find Slack users by display name, Slack ID, email, or linked Operator username |
| `slack_list_channels(query="")` | list postable channels from the current Slack snapshot |
| `slack_read_channel(channel, count=20)` | fetch recent channel messages |
| `slack_read_thread(channel, thread_id, count=50)` | fetch replies from a thread |
| `slack_add_reaction(channel, message_id, emoji)` | add a reaction |
| `slack_remove_reaction(channel, message_id, emoji)` | remove a reaction |

The built-in `send_message` tool also works with Slack channel names like `#general` or raw channel IDs.

Mention guidance:

- use `@Name` only when the display name is unambiguous
- when in doubt, call `slack_find_users` and use the returned explicit `<@UID>` mention

## Stop And Busy Handling

If a message is exactly `stop` or `cancel`, Operator handles it directly.

That message:

- does not go to the model
- cancels the active request for that conversation if one exists

If the conversation is active, Operator first replies:

```text
Cancelling...
```

and then, once the task has been cancelled:

```text
Request stopped.
```

If the conversation is idle, Operator replies:

```text
No active request to stop.
```

If a second request arrives while the same conversation is still running, Operator replies:

```text
Still processing a request. Say `stop` to stop it.
```

## File Handling

Inbound Slack files are downloaded into the active agent workspace under `workspace/inbox/`.

Behavior by file type:

| File type | Behavior |
| --- | --- |
| images up to 5 MB (`png`, `jpeg`, `gif`, `webp`) | saved to `inbox/` and also inlined to the model as image content |
| other files up to 50 MB | saved to `inbox/` and described in text |
| larger than 50 MB | skipped |

Additional details:

- duplicate filenames get suffixed
- transport downloads over 50 MB are rejected
- outbound uploads use Slack `files.upload_v2`

## Snapshot Refresh

Operator refreshes the Slack channel snapshot on startup and when the subscribed channel lifecycle events arrive.

Operator refreshes the user snapshot on startup and when the subscribed user lifecycle events arrive.

## Common Debugging Checks

```sh
operator logs -n 100
operator config
operator agents
```

Then verify:

- the Slack app is installed to the workspace
- the bot is invited to the channel
- your Slack user is linked with `operator user add` or `operator user link`
- the agent transport points at the right env vars
