Operator uses Slack Socket Mode. You do not need a public webhook URL.

## Scan First

- Supported inbound patterns: DMs and `@mentions` in invited channels
- Supported files: inline images up to 5 MB, all other files saved locally up to 50 MB
- The runtime uses the authenticated user's stored timezone when rendering live prompt timestamps
- Slack adds transport tools like `slack_find_users`, `slack_list_channels`, and `slack_read_thread`
- Exact `stop` and `cancel` messages bypass the LLM and cancel the active request in that conversation

## Slack App Setup

### 1. Create the app

Create a Slack app and enable **Socket Mode**.

Create an app-level token with:

- `connections:write`

That becomes your `SLACK_APP_TOKEN`.

### 2. Add bot scopes

Under **OAuth & Permissions**, add:

- `app_mentions:read`
- `channels:history`
- `channels:read`
- `chat:write`
- `files:read`
- `files:write`
- `groups:history`
- `groups:read`
- `im:history`
- `im:read`
- `im:write`
- `reactions:write`
- `users:read`

### 3. Subscribe to events

Subscribe to:

- `app_mention`
- `message.im`
- `reaction_added`
- `reaction_removed`

That covers:

- DMs
- channel mentions
- reaction system events

### 4. Install the app

Install the app to your workspace and copy the bot token:

- `SLACK_BOT_TOKEN`

### 5. Configure Operator

Add the tokens to `~/.operator/.env`:

```sh
SLACK_BOT_TOKEN=xoxb-...
SLACK_APP_TOKEN=xapp-...
```

Then configure the agent transport:

```yaml
agents:
  operator:
    transport:
      type: slack
      env:
        bot_token: SLACK_BOT_TOKEN
        app_token: SLACK_APP_TOKEN
      settings:
        include_archived_channels: false
        inject_channels_into_prompt: true
        inject_users_into_prompt: true
        expand_mentions: true
```

## What The Runtime Handles

Slack transport accepts:

- DMs from human users
- `@mentions` in channels where the bot is invited

It ignores:

- bot-authored messages
- most Slack system and edit subtypes

## Channel, User, and Thread Context

Slack runtime behavior includes:

- treating Slack conversations as thread-scoped sessions
- resolving user IDs to human names
- resolving channel IDs to `#channel-name`
- injecting the current user list and current channel snapshot into the system prompt when enabled
- exposing message metadata (`message_id`, `channel_id`, and `thread_id`) as a request-only context block

Slack session rules are intentional:

- a top-level DM starts a fresh private session thread
- a top-level channel `@mention` starts a fresh channel session thread
- in channels, later replies continue that session only when they mention the agent again
- unmentioned human replies in the thread are ambient chatter and are ignored unless the agent inspects them with `slack_read_thread(...)`

The injected channel list is refreshed on startup and on Slack channel lifecycle events such as create, rename, archive, and unarchive. Old channel names do not remain as prompt-visible aliases after a refresh.

When the agent is first mentioned inside an existing thread, Operator may inject a one-time snapshot of earlier replies in that thread so the model has context it missed before being added.

Reaction events are buffered as lightweight system events and injected into the next request in the same conversation.

## Stop And Cancel

Slack does not have a general in-chat command surface.

If a message contains only `stop` or `cancel` after trimming whitespace and lowercasing, the runtime handles it directly before prompt assembly or model execution.

- the active request in that conversation is cancelled
- the control message is not sent to the model
- any other text, including `please stop`, is treated as a normal user message

If there is no active request in that conversation, Operator replies:

```text
No active request to stop.
```

## Conversation Behavior

Slack conversation IDs are built from:

```text
slack:<agent-name>:<channel-id>:<root-thread-ts>
```

That means the Slack thread root is the conversation boundary. New top-level DMs and new top-level channel mentions start fresh sessions. Replies stay attached to that root thread, but channel replies still need to mention the agent to count as new input.

If a second message arrives while the same conversation is still busy, Operator replies:

```text
Still processing a request. Say `stop` to stop it.
```

## Slack Transport Tools

These tools exist only during live Slack-backed runs:

| Tool | Use it for |
| --- | --- |
| `slack_find_users(query="", limit=20, linked_only=false)` | find Slack users by display name, handle, Slack ID, email, or linked Operator username; use the returned `<@UID>` mention when names are ambiguous |
| `slack_list_channels(query="")` | list postable channels, IDs, and current topic or purpose snippets from the refreshed snapshot |
| `slack_read_channel(channel, count=20)` | fetch recent channel messages |
| `slack_read_thread(channel, thread_id, count=50)` | fetch thread replies |
| `slack_add_reaction(channel, message_id, emoji)` | add an emoji reaction |
| `slack_remove_reaction(channel, message_id, emoji)` | remove an emoji reaction |

`send_message` also accepts Slack channel names like `#general` or raw channel IDs.
Raw `@Name` only auto-expands when the display name is unique. When multiple Slack users share a display name, use the explicit `<@UID>` mention returned by `slack_find_users(...)`.

## File Handling

Inbound Slack files behave like this:

| File type | Runtime behavior |
| --- | --- |
| image/png, image/jpeg, image/gif, image/webp up to 5 MB | inlined into the model request as an image block and saved under `workspace/inbox/` |
| other file types up to 50 MB | saved under `workspace/inbox/` and described in text |
| larger than 50 MB | skipped |

Additional details:

- duplicate filenames get suffixed (`file_1.txt`, etc.)
- unsafe filenames are sanitized to a basename
- saved files can be read later with workspace tools

Outbound uploads use `send_file`, which calls Slack `files.upload_v2`.
