---
title: "Skills"
description: "Reusable instruction modules with optional references, assets, and scripts"
---

Skills are reusable know-how. They teach an agent how to do a class of work without hard-coding every detail into `AGENT.md`.

Each skill lives at `~/.operator/skills/<name>/SKILL.md`.

## Why Skills Exist

Use a skill when the knowledge is:

- reusable across agents
- specific enough that it would bloat the base prompt
- better taught as instructions plus references or scripts

Good examples:

- PR review process
- docs publishing workflow
- a house style for release notes
- an internal deployment procedure

## Directory Layout

```text
~/.operator/skills/
└── docs-auditor/
    ├── SKILL.md
    ├── scripts/
    ├── references/
    └── assets/
```

Only `SKILL.md` is required.

## `SKILL.md`

```yaml
---
name: docs-auditor
description: Reviews documentation for accuracy against the codebase and rewrites outdated pages.
metadata:
  env:
    - GITHUB_TOKEN
---

# Docs Auditor

1. Inspect the relevant code paths first.
2. Treat the code as the source of truth.
3. Fix inaccurate docs before polishing wording.
4. Prefer small, example-driven guides over vague prose.
```

Frontmatter is what the runtime uses for discovery:

- `name`
- `description`
- optional `metadata.env`

## Discovery Model

Skills are loaded progressively:

1. the system prompt gets only skill name and description
2. the agent calls `read_skill(...)` when it needs the skill body or a reference file
3. the agent calls `run_skill(...)` when the skill includes scripts or commands

Why it works this way:

- the base prompt stays small
- detailed instructions are available on demand
- large references do not pollute every conversation

## Skill Permissions

Skill access is separate from tool access.

```yaml
permissions:
  skills: [docs-auditor]
```

That means:

- an agent may be allowed to see only one skill
- an agent may be able to read a skill but still lack unrelated tools
- `skills: "*"` grants discovery of every skill

## `read_skill`

`read_skill(skill, path="")` reads:

- `SKILL.md` when `path` is empty
- a specific file inside the skill directory when `path` is set

Path traversal is blocked, so the call cannot escape the skill directory.

## `run_skill`

`run_skill(skill, command, timeout=120)` executes a command in skill context.

Important runtime behavior:

- command strings are parsed with `shlex`
- execution uses `shell=False`
- pipes, redirects, and chaining are not available unless you explicitly invoke a shell yourself
- subprocess `cwd` is the current agent workspace
- `SKILL_DIR` is set automatically
- arguments starting with `scripts/`, `references/`, or `assets/` are expanded to absolute paths inside the skill

Security and hygiene details:

- transport secret env vars are stripped before execution
- most `OPERATOR_*` vars are stripped
- `$OPERATOR_HOME` is preserved
- `$VAR` and `${VAR}` references are expanded after sanitization

That keeps skill scripts useful without quietly inheriting Slack tokens or unstable runtime-only state.

## Environment Variables

Declare required env vars in frontmatter:

```yaml
metadata:
  env:
    - GITHUB_TOKEN
    - STRIPE_API_KEY
```

Missing env vars appear in:

- `operator skills`
- `operator skills list`
- the skill metadata block injected into the agent prompt

## Creating And Updating Skills

You have two supported paths:

1. edit files directly in `~/.operator/skills/`
2. use the deterministic tools `create_skill`, `update_skill`, `delete_skill`, and `list_skills`

The tool-based path enforces:

- lowercase alphanumeric plus hyphen skill names
- max 64 characters
- required description
- non-empty instructions

It also warns when the body is longer than 500 lines, which is usually a sign that content should be split into `references/` files instead.

## Authoring Guidance

Write skills the same way you would brief a capable teammate:

- explain when to use the skill
- define the order of operations
- include concrete examples
- keep rare edge-case detail in `references/`
- put executable helpers in `scripts/`

Use `$OPERATOR_HOME` inside scripts or instructions instead of hard-coding `~/.operator`. It works in more contexts and matches the runtime environment.
