title: "Skills"

Skills are reusable capabilities stored under `~/.operator/skills/<name>/SKILL.md`.

They are not tools. A skill teaches an agent how to do a class of work. The agent still chooses which tools to use while following that skill.

## What A Skill Looks Like

Example `SKILL.md`:

```yaml
---
name: docs-auditor
description: Audits a docs site against the current source code and rewrites stale pages.
metadata:
  env:
    - GITHUB_TOKEN
---

# Docs Auditor

## When To Use

Use when the user asks to compare docs against code and bring them back into sync.

## Workflow

1. Read the source of truth first.
2. Identify the public surface area.
3. Rewrite the docs in the order a new user would need them.
```

## How Skills Are Discovered

Operator scans `~/.operator/skills/` and reads each `SKILL.md` frontmatter.

Discovery uses:

- `name`
- `description`
- optional `metadata.env`

If required env vars are missing, the skill is still listed, but marked as missing env.

## Create Skills Through Tools

Agents can manage skills with typed tools:

- `create_skill`
- `update_skill`
- `delete_skill`
- `list_skills`

Example:

```python
create_skill(
    name="pr-reviewer",
    description="Reviews pull requests and reports logic, safety, and test coverage issues.",
    instructions="# PR Reviewer\n\n1. Fetch the diff\n2. Review for risks\n3. Return findings first",
    env="GITHUB_TOKEN",
)
```

Name rules match the current validator:

- lowercase alphanumeric plus hyphens
- 1 to 64 characters
- no leading or trailing hyphen
- no consecutive hyphens

## Reading And Running Skills

Operator also exposes skill access tools:

- `read_skill(skill, path="")`
- `run_skill(skill, command, timeout=120)`

Use `read_skill` first when the skill has references, scripts, or assets you need to inspect.

`run_skill` is for deterministic execution in skill context, such as:

- running a helper script from `scripts/`
- using a checked-in reference file from `references/`
- invoking an asset pipeline from `assets/`

## Skill Directory Structure

A good skill keeps `SKILL.md` short and moves bulk content into subdirectories:

```text
skills/<name>/
├── SKILL.md
├── scripts/
├── references/
└── assets/
```

Why this is useful:

- `SKILL.md` stays readable
- references can grow without bloating prompt context
- scripts stay executable and versioned with the skill

## How `run_skill` Works

Technical details from the implementation:

- the command string is parsed with `shlex.split`
- the process runs with `shell=False`
- `scripts/`, `references/`, and `assets/` path prefixes are expanded to absolute skill paths
- the process runs in the current agent workspace
- `SKILL_DIR` is set in the environment
- transport secret env vars are stripped
- `OPERATOR_*` env vars are stripped except `OPERATOR_HOME`

That design makes skill scripts portable while avoiding accidental secret leakage.

## Good Skill Boundaries

Use a skill when the reusable part is instruction and procedure.

Use an agent when the reusable part is identity and permissions.

Use memory when the reusable part is durable knowledge.

Use a tool when the reusable part is executable behavior.

Keeping those separate is what makes Operator understandable at scale.
