Skills live in `~/.operator/skills/<name>/` and are described by `SKILL.md`.

## Scan First

- Operator follows the agentskills.io frontmatter rules.
- At startup, only skill metadata is injected into the prompt.
- Agents use `read_skill` for progressive disclosure and `run_skill` for skill scripts.
- Missing bundled skills are auto-installed on `operator init` and runtime startup.
- Skill permissions are separate from tool permissions.

## Directory Layout

```text
~/.operator/skills/
└── my-skill/
    ├── SKILL.md
    ├── scripts/
    ├── references/
    └── assets/
```

Only `SKILL.md` is required.

## `SKILL.md`

```yaml
---
name: pr-reviewer
description: Reviews pull requests and highlights risks, regressions, and missing tests.
metadata:
  env:
    - GITHUB_TOKEN
---

# PR Reviewer

1. Read the PR description.
2. Inspect the diff.
3. Prioritize correctness, security, behavior changes, and missing tests.
4. Return findings first, then a short summary.
```

### Frontmatter Rules

| Field | Required | Rules |
| --- | --- | --- |
| `name` | yes | lowercase letters, numbers, and hyphens only; must match the directory name; max 64 chars |
| `description` | yes | max 1024 chars |
| `metadata.env` | no | string or list of required env var names |

The body must not be empty. Operator also warns when the body exceeds 500 lines and recommends moving detailed material into `references/`.

## Progressive Disclosure

Skills are intentionally loaded in layers:

1. **Discovery:** only `name` and `description` are shown in the system prompt
2. **Activation:** the agent reads `SKILL.md`
3. **Deep reference:** the agent reads specific `references/` or `assets/` files only when needed

This keeps the prompt surface small while still supporting detailed skills.

## `read_skill` and `run_skill`

### `read_skill`

Use `read_skill(skill, path="")` to read:

- `SKILL.md`
- files under `references/`
- files under `assets/`

Path traversal is blocked.

### `run_skill`

Use `run_skill(skill, command, timeout=120)` when the skill includes scripts or command examples.

Important behavior:

- command strings are parsed with `shlex`
- execution uses `shell=False`
- pipes, redirects, and command chaining are not available unless you explicitly invoke a shell yourself
- subprocess `cwd` is the agent workspace, not the skill directory
- `SKILL_DIR` is set automatically
- arguments starting with `scripts/`, `references/`, or `assets/` are expanded to absolute paths inside the skill
- transport token env vars and most `OPERATOR_*` vars are stripped before execution
- `$OPERATOR_HOME` is preserved

## Environment Variables

Declare required variables in frontmatter:

```yaml
metadata:
  env:
    - GITHUB_TOKEN
    - STRIPE_API_KEY
```

Operator surfaces missing env vars in:

- `operator skills`
- `operator skills list`
- the injected skills prompt shown to agents

## Bundled Skills

Operator currently ships with:

| Skill | Purpose |
| --- | --- |
| `job-creator` | teaches agents how to create and manage scheduled jobs |
| `skill-creator` | teaches agents how to create and refine skills themselves |

You can restore bundled skill directories with the [Skills CLI](/cli/skills).

## Permissions

```yaml
agents:
  operator:
    permissions:
      skills: [job-creator, skill-creator]
```

If `permissions.skills` is set, only the listed skills are available to that agent.

This is independent from tool permissions. An agent might be allowed to see a skill but still lack `run_shell`, in which case `run_skill` can still be the right interface.

## Creating and Updating Skills

You have three supported paths:

1. edit files directly in `~/.operator/skills/`
2. use the `manage_skill` tool in a live agent run
3. restore bundled skills with `operator skills reset`

The CLI does **not** create or update arbitrary skills. It only lists skills and resets bundled ones.

## Path Guidance

Use `$OPERATOR_HOME`, not `~/.operator`, inside skill instructions and scripts.

```text
$OPERATOR_HOME/skills/my-skill/scripts/run.py
```

That works in shell and non-shell contexts.
