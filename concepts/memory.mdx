title: "Memory"
description: "File-backed rules, searchable notes, scope rules, expiry, and indexing"
---

Operator memory is file-backed. Markdown files are the source of truth, and SQLite is used only for search and indexing.

## The Core Split

Memory has two behaviors:

| Kind | Behavior | Best for |
| --- | --- | --- |
| `rules/` | always injected into the prompt | short standing guidance |
| `notes/` | searched on demand | facts, references, and durable context |

That split is deliberate. Rules should stay small and high-signal. Notes can be broader because they are not injected every turn.

## Directory Layout

```text
~/.operator/
├── memory/
│   ├── global/
│   │   ├── rules/
│   │   ├── notes/
│   │   └── trash/
│   └── users/
│       └── <user>/
│           ├── rules/
│           ├── notes/
│           └── trash/
└── agents/
    └── <agent>/
        └── memory/
            ├── rules/
            ├── notes/
            └── trash/
```

Each memory file is markdown with minimal frontmatter:

- `created_at`
- `updated_at`
- optional `expires_at`

## Scopes

The tool-facing scopes are:

- `agent`
- `user`
- `global`

How they resolve:

| Scope | Backing path | Available in chat | Available in jobs |
| --- | --- | --- | --- |
| `agent` | `agents/<agent>/memory/` | yes | yes |
| `user` | `memory/users/<username>/` | private chats only | no |
| `global` | `memory/global/` | yes | yes |

Why `user` scope is restricted:

- it depends on an authenticated human identity
- public channels should not quietly pull private user memory
- jobs do not run as a user

## Saving Memory

Rules and notes are stored by deterministic key.

```text
save_rule("response-style", "Lead with findings and keep summaries short.", scope="agent")
save_note("release-window", "The next release goes out Fridays at 16:00 UTC.", scope="global")
save_note("follow-up", "Ask again next week.", scope="user", ttl="7d")
```

Saving with the same key replaces the file in place and updates timestamps.

TTL formats for notes:

- `30m`
- `1h`
- `3d`
- `2w`

## Reading And Searching

Available memory tools:

- `save_rule`
- `save_note`
- `search_notes`
- `list_rules`
- `list_notes`
- `read_note`
- `forget_rule`
- `forget_note`

Notes are searched through the SQLite index at `~/.operator/db/memory_index.db`.

Search behavior:

- FTS5 with Porter stemming is always available
- the runtime reindexes changed files on startup
- manual file edits should be followed by `operator memory index`

If embeddings are configured and `sqlite-vec` is installed, note search also stores vectors for semantic matching.

## Forgetting And Expiry

Operator does not hard-delete active memory by default.

- `forget_rule` moves a rule file to `trash/`
- `forget_note` moves a note file to `trash/`
- expired notes are swept into `trash/` automatically by the background runtime once per hour

This makes memory safer to debug because you can still inspect what was removed.

## How Rules Reach The Prompt

Prompt assembly injects rules in this order:

1. global rules
2. agent rules
3. user rules when the run is private and user-scoped

Notes are never blindly injected. The agent must search for them.

That keeps the prompt smaller and forces durable context to stay explicit.

## Practical Guidance

Use:

- rules for stable style or policy constraints
- notes for facts that may need lookup later
- state for machine bookkeeping such as cursors and queues

If a fact should shape every answer, it probably belongs in a rule.

If a fact is only sometimes relevant, it probably belongs in a note.

## Debugging

Start with the CLI:

```sh
operator memory list agent:operator
operator memory search "release schedule" --scope agent:operator
operator memory index
```

Then inspect the files directly under:

- `~/.operator/agents/<agent>/memory/`
- `~/.operator/memory/global/`
- `~/.operator/memory/users/<user>/`
