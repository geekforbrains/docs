Operator memory is backed by SQLite plus `sqlite-vec`.

## Scan First

- Runtime memory is only active when `memory.harvester.enabled` or `memory.cleaner.enabled` is `true`.
- Automatic recall happens per live request, not by mutating stored history.
- `user` scope is available only in private chat contexts.
- Jobs use `agent` and `global` memory only.
- `candidate` and `durable` are retention classes; `pinned` is an injection policy.

<Warning>
  If both `memory.harvester.enabled` and `memory.cleaner.enabled` are `false`, the runtime does not configure memory tools or semantic recall. The `operator memories` CLI still reads the stored memory tables directly.
</Warning>

## How Recall Works

For a chat turn with memory enabled:

1. Operator embeds the latest user message.
2. It searches the allowed scopes.
3. Results are filtered by `memory.inject_min_relevance`.
4. The best matches up to `memory.inject_top_k` are injected as a request-only context snapshot.
5. Pinned memories are appended separately to the system prompt.

This injected material is not written back into the persisted conversation.

## Scopes

| Scope | Used in chat | Used in jobs | Notes |
| --- | --- | --- | --- |
| `user` | private chats only | no | tied to the authenticated user context for the run |
| `agent` | yes | yes | scoped to the current agent |
| `global` | yes | yes | shared across agents |

Public channels never expose `user` scope.

## Retention vs Pinning

| Property | Meaning | Expiry |
| --- | --- | --- |
| `candidate` | short-lived but reusable context | expires after `candidate_ttl_days` |
| `durable` | long-lived reusable fact | does not auto-expire |
| `pinned` | always inject this memory | does not auto-expire |

Pinned memories are separate from retention. A pinned durable memory stays durable. A pinned candidate memory is effectively preserved because pinning disables expiry.

## Saving Behavior

When memory is enabled, `save_memory(...)`:

- embeds the content
- deduplicates against the closest existing memory in the same scope
- merges pin/retention state when a near-duplicate already exists
- enforces `max_memories` per scope

That means repeated saves of essentially the same fact often update an existing row instead of creating a new one.

## Harvester

The harvester is a scheduled LLM worker.

It:

- scans recently updated conversations
- looks only at user and assistant text
- extracts reusable facts
- classifies them as `candidate` or `durable`
- ignores weak or obviously ephemeral information

It is intentionally conservative and prefers extracting nothing over storing noise.

## Cleaner

The cleaner is another scheduled LLM worker.

It processes scopes independently and may:

- rewrite wording for clarity
- split compound memories into separate facts
- merge duplicates
- delete stale or contradicted memories

Pinned memories may not be deleted by the cleaner.

## Good vs Bad Memories

Good durable memories:

- stable user preferences
- standing team conventions
- reusable project facts
- long-lived agent instructions

Good candidate memories:

- temporary but reusable context likely to matter again soon

Bad memories:

- one-off tasks
- current troubleshooting state
- branch names
- meeting logistics
- “right now” status updates

## Tool Surface

When runtime memory is enabled, agents can use:

| Tool | Use it for |
| --- | --- |
| `save_memory` | store a fact with scope and retention |
| `search_memories` | search semantically relevant memories |
| `forget_memory` | delete a specific memory |
| `list_memories` | list currently visible memories |

Example calls:

```text
save_memory("Use Slack thread replies for daily summaries", scope="agent", retention="durable")
save_memory("Release checklist is active this week", scope="agent", retention="candidate")
save_memory("Always include links in the handoff", scope="agent", retention="durable", pinned=true)
```

If runtime memory is disabled, those tools still exist in the tool registry but return an error like `memory system not configured`.

## Configuration Knobs

```yaml
memory:
  embed_model: "openai/text-embedding-3-small"
  embed_dimensions: 1536
  max_memories: 10000
  inject_top_k: 5
  inject_min_relevance: 0.1
  candidate_ttl_days: 14
  harvester:
    enabled: true
    schedule: "*/30 * * * *"
    model: "openai/gpt-4.1-mini"
  cleaner:
    enabled: true
    schedule: "0 3 * * *"
    model: "openai/gpt-4.1-mini"
```

Tuning guidelines:

- lower `inject_top_k` if too much memory context is injected
- raise `inject_min_relevance` if recall feels noisy
- raise `candidate_ttl_days` if candidate memories expire too quickly

## Debugging

Start with the CLI:

```sh
operator memories --scope agent --scope-id operator
operator memories stats
```

Then inspect logs:

```sh
operator logs -f | rg 'memory recall:|memory injection:'
```

Typical log lines:

```text
INFO  [operator:abc123] memory recall: requested_top_k=5 raw_matches=4 filtered_matches=2 returned=2 min_relevance=0.100
INFO  [operator:abc123] memory injection: injected=2 retention_mix=candidate=1 durable=1
```

Those tell you:

- how many matches vector search found
- how many survived the relevance filter
- how many were actually injected
- whether the injected mix skewed candidate or durable
