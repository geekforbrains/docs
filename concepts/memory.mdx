Operator memory is file-backed. Markdown files are the source of truth, and SQLite is used only for search indexing.

## Scan First

- Memory lives in files under `~/.operator/memory/` and `~/.operator/agents/<name>/memory/`.
- Rules are always injected into the system prompt.
- Notes are searched on demand.
- `user` scope is available only in private chat contexts.
- Jobs use `agent` and `global` memory only.
- Expired notes and forgotten items move to `trash/` instead of being hard-deleted.

## Directory Layout

```text
~/.operator/
├── memory/
│   ├── global/
│   │   ├── rules/
│   │   ├── notes/
│   │   └── trash/
│   └── users/
│       └── <user>/
│           ├── rules/
│           ├── notes/
│           └── trash/
└── agents/
    └── <agent>/
        └── memory/
            ├── rules/
            ├── notes/
            └── trash/
```

Each memory file is markdown with minimal frontmatter:

- `created_at`
- `updated_at`
- optional `expires_at`

## Rules vs Notes

| Kind | Behavior | Best for |
| --- | --- | --- |
| `rules/` | always injected | short standing instructions and durable conventions |
| `notes/` | searched on demand | facts, references, and reusable context that should not bloat every prompt |

Rules do not have TTLs. Notes can optionally expire.

## Scopes

| Scope | Used in chat | Used in jobs | Notes |
| --- | --- | --- | --- |
| `user` | private chats only | no | tied to the authenticated user context for the run |
| `agent` | yes | yes | scoped to the current agent |
| `global` | yes | yes | shared across agents |

Public channels never expose `user` scope.

## Saving Behavior

Memory tools use deterministic keys. Saving to an existing key replaces the file in place and updates its timestamps.

### `save_rule`

```text
save_rule("response-style", "Lead with findings and keep summaries short.", scope="agent")
```

Use rules for short guidance that should shape every interaction.

### `save_note`

```text
save_note("release-window", "The next release goes out on Fridays at 16:00 UTC.", scope="agent")
save_note("launch-checklist", "Reminder to post to #announce after deploy.", scope="user", ttl="7d")
```

Notes support optional TTLs:

- `30m`
- `1h`
- `3d`
- `2w`

## Search

`search_notes(query, scope="agent")` searches note files through the SQLite index at `~/.operator/db/memory_index.db`.

Search behavior:

- FTS5 is always used for text search
- if `defaults.embeddings` is configured and `sqlite-vec` is installed, Operator also stores vectors for semantic retrieval
- the index is rebuilt from files, so the files remain the source of truth

## Forgetting And Expiry

These tools move memory files to `trash/`:

- `forget_rule(key, scope="...")`
- `forget_note(key, scope="...")`

Expired notes are also swept to `trash/` automatically by the background runtime once per hour.

## Tool Surface

Agents can use:

| Tool | Use it for |
| --- | --- |
| `save_rule` | create or replace a standing rule by key |
| `save_note` | create or replace a searchable note by key |
| `search_notes` | search notes by key and content |
| `list_rules` | list active rules in a scope |
| `list_notes` | list active notes in a scope |
| `read_note` | read the full content of a note |
| `forget_rule` | move a rule to trash |
| `forget_note` | move a note to trash |

Example calls:

```text
save_rule("daily-summary-style", "Use a short teaser plus a threaded deep dive.", scope="agent")
save_note("slack-app-id", "A1234567890", scope="global")
search_notes("release schedule", scope="agent")
list_rules(scope="agent")
read_note("slack-app-id", scope="global")
```

## Debugging

Start with the CLI:

```sh
operator memory list agent:operator
operator memory search "release schedule" --scope agent:operator
operator memory index
```

Then inspect the files directly:

```text
~/.operator/agents/operator/memory/
~/.operator/memory/global/
~/.operator/memory/users/<user>/
```

If you edit memory files by hand, run `operator memory index` so the search index catches up.
