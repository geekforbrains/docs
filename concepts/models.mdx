Operator uses LiteLLM model IDs like `anthropic/claude-sonnet-4-6` or `openai/gpt-4.1`.

## Scan First

- `models:` is a fallback chain, not a single preferred value.
- Fallback happens on every live model call.
- `thinking:` is `off|low|medium|high`.
- `context_ratio` budgets the full outbound request shape, including tool schemas.
- Jobs inherit `thinking`, `context_ratio`, and `max_output_tokens` from the resolved agent.
- Jobs may override `model` and `max_iterations`, but not `thinking`.

## Fallback Chains

```yaml
defaults:
  models:
    - "anthropic/claude-sonnet-4-6"
    - "openai/gpt-4.1"
```

Behavior:

1. Operator tries the first model.
2. If the call errors or returns an unusable response, it retries the same request with the next model.
3. If a later model succeeds, the run continues normally.

Fallback is evaluated per live call, not per conversation. A long conversation may use different models on different turns if failures occur.

An unusable response means the provider returned a syntactically successful result that does not contain the required completion payload, for example an empty `choices` list.

## Thinking Levels

```yaml
defaults:
  thinking: off

agents:
  researcher:
    models:
      - "openai/o3"
    thinking: high
```

Supported values:

- `off`
- `low`
- `medium`
- `high`

Operator maps these to LiteLLM reasoning controls when the current model supports them:

| Operator | LiteLLM |
| --- | --- |
| `off` | no reasoning request; usually `reasoning_effort="none"`, but omitted for Anthropic compatibility |
| `low` | `reasoning_effort="low"` |
| `medium` | `reasoning_effort="medium"` |
| `high` | `reasoning_effort="high"` |

If the current model does not support reasoning controls, Operator omits the parameter and continues.

## Cross-Provider Safety

When a response includes provider-specific reasoning metadata, Operator strips that metadata from assistant history before the next model call. This keeps Anthropic-to-OpenAI or OpenAI-to-other-provider fallbacks from breaking on incompatible history payloads.

## Inheritance Rules

| Execution type | Models | Thinking | Other execution settings |
| --- | --- | --- | --- |
| Chat | agent override or `defaults.models` | agent override or `defaults.thinking` | `max_iterations`, `context_ratio`, and `max_output_tokens` resolve the same way |
| `spawn_agent()` without `agent=` | current agent models | current agent thinking | fresh child run of the current agent; preserves the current run mode and session/job envelope, but not parent conversation history |
| `spawn_agent(agent="other")` | target agent models | target agent thinking | fresh child run of the target agent with that agent's own prompt, memory, state, skills, workspace, and permissions; preserves the current run mode and session/job envelope, but not parent conversation history |
| Scheduled job | `job.model` or resolved agent models | resolved agent thinking | resolved agent `context_ratio` and `max_output_tokens` |

## Jobs

Job frontmatter supports:

- `model`
- `max_iterations`

It does **not** support:

- `thinking`
- `context_ratio`
- `max_output_tokens`

If you need a different thinking level for a job, point the job at a different agent.

## `max_output_tokens`

`max_output_tokens` sets LiteLLM `max_tokens` for each call.

- if configured: Operator uses your explicit value
- if `null`: Operator asks LiteLLM for the model's max output size and uses that when available
- if LiteLLM cannot resolve a max: the parameter is omitted

## `context_ratio`

`context_ratio` controls how aggressively Operator trims the live request before each model call.

- higher values keep more history
- lower values reduce token usage
- the budget is measured against the full outbound payload: system prompt, conversation messages, multimodal content, and tool schemas
- the value must be greater than `0` and at most `1`

The system prompt and latest user exchange are preserved first; older exchange groups are the first thing dropped.
