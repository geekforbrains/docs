Operator uses LiteLLM model IDs such as `anthropic/claude-sonnet-4-6` or `openai/gpt-4.1`.

## Fallback Chains

```yaml
defaults:
  models:
    - "anthropic/claude-sonnet-4-6"
    - "openai/gpt-4.1"
```

`models` is a fallback chain, not just a preference list.

Per model call, Operator:

1. tries the first model
2. retries the same request with the next model if the current one errors or returns an unusable response
3. continues with the first model that returns a valid message

Fallback happens per live call, not per conversation. One conversation can use different providers on different turns if failures occur.

## Thinking Levels

```yaml
defaults:
  thinking: off
```

Supported values:

- `off`
- `low`
- `medium`
- `high`

Operator maps these to provider reasoning controls when the model supports them. When the model does not support reasoning control, Operator drops the provider-specific parameter and continues.

Important implementation detail:

- for Anthropic models, `thinking: off` omits `reasoning_effort` entirely for compatibility
- for other supported models, `off` generally maps to `reasoning_effort="none"`

## OpenAI Responses Bridge

When a model both:

- supports reasoning controls
- and the run includes tools

Operator may switch OpenAI requests onto LiteLLM's Responses bridge automatically.

Why this exists:

- it keeps tool use and reasoning control working together more reliably on supported OpenAI models

You still configure the model normally. The bridge selection is internal request logic.

## Inheritance Rules

| Execution type | Models | Thinking | Other execution settings |
| --- | --- | --- | --- |
| Chat run | agent override or `defaults.models` | agent override or `defaults.thinking` | agent override or defaults |
| `spawn_agent()` without `agent=` | current agent | current agent | current agent |
| `spawn_agent(agent="other")` | target agent | target agent | target agent |
| Job run | `job.model` or resolved agent models | resolved agent thinking | resolved agent settings, except `job.max_iterations` may override |

Jobs may override:

- `model`
- `max_iterations`

Jobs do not override:

- `thinking`
- `context_ratio`
- `max_output_tokens`

## `max_output_tokens`

`max_output_tokens` controls the per-call `max_tokens` value sent through LiteLLM.

Behavior:

- if you set it explicitly, Operator uses your value
- if it is `null`, Operator asks LiteLLM for the model's max output size
- if LiteLLM cannot resolve a max, Operator omits the parameter

## `context_ratio`

`context_ratio` controls how much of the model window Operator is willing to spend on live request context before each model call.

That budget includes:

- conversation history
- multimodal content
- tool schemas
- the rest of the outbound request payload

When the request is too large, Operator preserves:

- the system prompt
- the newest exchange groups

Then it trims older conversation history first.

## Cross-Provider History Safety

When provider-specific reasoning metadata appears in assistant history, Operator strips that provider-specific content before the next call.

Why it matters:

- Anthropic-to-OpenAI or OpenAI-to-other-provider fallback should not break because one provider stored incompatible reasoning blocks in history

This is one of the pieces that makes cross-provider fallback workable in longer runs.
