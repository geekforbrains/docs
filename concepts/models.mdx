Operator uses LiteLLM model IDs such as `anthropic/claude-sonnet-4-6` or `openai/gpt-4.1`.

This page covers the runtime behavior around those models, not just the config fields.

## Fallback Chains

```yaml
defaults:
  models:
    - "anthropic/claude-sonnet-4-6"
    - "openai/gpt-4.1"
```

`models` is a fallback chain, not just a preference list.

Per model call, Operator:

1. tries the first model
2. retries the same request with the next model if the current one errors or returns an unusable response
3. continues with the first model that returns a valid message

Fallback happens per live call, not per conversation.

## Thinking Levels

```yaml
defaults:
  thinking: off
```

Supported values:

- `off`
- `low`
- `medium`
- `high`

Operator maps these to provider reasoning controls when the concrete model supports them.

Important behavior from the implementation:

- Anthropic with `thinking: off` omits `reasoning_effort` entirely for compatibility
- when a model supports reasoning control, Operator maps the level to `reasoning_effort`
- when a model does not support reasoning control, Operator logs and continues without it

## OpenAI Responses Bridge

When all of these are true:

- the request has tools
- the model supports reasoning control
- the provider resolves to OpenAI

Operator routes the request through LiteLLM's Responses bridge automatically.

You still configure the model normally. The bridge selection is runtime behavior, not a different config shape.

## Job And Sub-Agent Resolution

| Execution type | Models | Thinking | Notes |
| --- | --- | --- | --- |
| chat run | agent override or `defaults.models` | agent override or `defaults.thinking` | normal agent inheritance |
| `spawn_agent()` without `agent=` | current agent | current agent | fresh run, same agent |
| `spawn_agent(agent="other")` | target agent | target agent | uses target agent config |
| job run | `job.model` or resolved agent models | resolved agent thinking | `job.max_iterations` may override |

Jobs may override:

- `model`
- `max_iterations`

Jobs do not override:

- `thinking`
- `context_ratio`
- `max_output_tokens`

## `max_output_tokens`

`max_output_tokens` controls the per-call `max_tokens` value sent through LiteLLM.

Behavior:

- if you set it explicitly, Operator uses your value
- if it is `null`, Operator asks LiteLLM for the model max output size when available
- if LiteLLM cannot resolve a max, Operator omits the parameter

## `context_ratio`

`context_ratio` controls how much of the model input budget Operator is willing to spend on live request context.

That budget includes:

- conversation history
- multimodal content
- tool schemas
- the rest of the outbound request payload

When the request is too large, Operator preserves:

- the system prompt
- the newest exchange groups

Then it trims older conversation history first.

## Tool Result Compression

Before older history is dropped for budget reasons, Operator also compresses old tool results.

Current behavior:

- the newest `5` tool results are kept as-is
- the next `10` are soft-trimmed to the start and end with a marker in the middle
- older tool results are replaced with a placeholder

This keeps long tool-heavy conversations from bloating context while preserving the most recent execution details.

## Time-Aware Message History

User messages, job prompts, and sub-agent task messages carry their creation time into model input.

They are rendered like:

```text
[Thursday, 2026-03-12T09:22:40-07:00]
```

If the user has a stored timezone, Operator renders timestamps in that timezone. Otherwise it falls back to UTC.

This keeps the prompt time-aware without mutating the stable system prompt.

## Cross-Provider History Safety

When provider-specific reasoning metadata appears in assistant history, Operator strips it before the next call.

Why this matters:

- fallback from one provider to another does not carry incompatible reasoning blocks forward
- longer conversations stay portable across providers

## Practical Defaults

For a first install:

- use one strong default model
- leave `thinking: off` until you have a specific need
- keep `max_iterations: 25`
- leave `max_output_tokens: null`
- change `context_ratio` only if you understand the tradeoff between history and tool/schema overhead
