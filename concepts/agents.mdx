Each agent is a named runtime identity from `operator.yaml`. The agent's human-authored prompt lives at `~/.operator/agents/<name>/AGENT.md`.

Use agents when you want different tool access, different models, different roles, or a different charter.

## Two Files Define An Agent

1. `operator.yaml` defines the runnable identity and execution settings.
2. `AGENT.md` defines the human-authored instructions.

Example config:

```yaml
agents:
  operator:
    permissions:
      tools: "*"
      skills: "*"
    transport:
      type: slack
      env:
        bot_token: SLACK_BOT_TOKEN
        app_token: SLACK_APP_TOKEN

  researcher:
    models:
      - "openai/o3"
    thinking: high
    permissions:
      tools: [read_file, write_file, list_files, web_fetch, read_skill, run_skill]
      skills: [docs-auditor]
```

Example `AGENT.md`:

```yaml
---
description: Deep research agent with web access.
---

You are a research specialist. Work methodically, cite sources in summaries,
and write long-form notes into the workspace when the task needs intermediate artifacts.
```

Important distinction:

- the configured key such as `researcher` is the real runtime name
- `AGENT.md` frontmatter contributes human-facing metadata, especially `description`

If `AGENT.md` is missing or malformed, the agent still exists because config is the source of truth.

## What An Agent Gets

Every agent has its own:

- `AGENT.md`
- workspace at `agents/<name>/workspace/`
- memory at `agents/<name>/memory/`
- state directory at `agents/<name>/state/`
- tool permission filter
- skill permission filter
- model and execution settings

The runtime also creates `workspace/shared` as a symlink to `~/.operator/shared/` so agents can exchange files without writing directly into each other's work directories.

## Workspace Layout

```text
agents/<name>/workspace/
├── inbox/
├── work/
├── artifacts/
├── tmp/
└── shared -> ~/.operator/shared/
```

Purpose of each directory:

- `inbox/`: inbound attachments and imported reference files
- `work/`: active drafts and intermediate files
- `artifacts/`: finished deliverables
- `tmp/`: disposable scratch files
- `shared/`: cross-agent file exchange

When Slack attachments arrive, Operator saves them into `inbox/`. Small images are also sent to the model as inline image content.

## Prompt Assembly

On each run, Operator builds the system prompt from broad to narrow:

1. `SYSTEM.md`
2. the active agent's `AGENT.md` body
3. discovered skill metadata the agent is allowed to see
4. configured peer-agent descriptions
5. transport or job context
6. active rules from memory

Why this matters:

- common behavior lives in `SYSTEM.md`
- agent-specific behavior lives in `AGENT.md`
- rules stay dynamic and editable without rewriting the base prompt
- the stable prefix stays cache-friendly while the dynamic suffix changes per run

If you edit `AGENT.md`, the next request uses the new prompt content. There is no special publish step.

## Permissions

Agent permissions are an allowlist.

- no `permissions` block means no tools and no skills
- `tools: "*"` means all built-in and runtime-added tools
- `skills: "*"` means all discovered skills
- lists restrict access to named tools or named skills

Example:

```yaml
permissions:
  tools: [read_file, write_file, list_files, web_fetch, spawn_agent]
  skills: [docs-auditor]
```

This separation is deliberate. Skills are reusable knowledge. Tools are executable capabilities.

## Headless Agents

An agent does not need a transport block to be useful.

Headless agents work well for:

- scheduled jobs
- `spawn_agent(...)` delegation
- specialist agents that should not be directly reachable by humans

Current runtime detail: the process still needs at least one transport somewhere in `operator.yaml` or startup exits. A headless agent is valid, but a fully headless deployment is not.

## Inheritance

Per-agent overrides are optional. When omitted, the agent inherits from `defaults`.

| Setting | Resolution |
| --- | --- |
| `models` | agent override or `defaults.models` |
| `thinking` | agent override or `defaults.thinking` |
| `max_iterations` | agent override or `defaults.max_iterations` |
| `context_ratio` | agent override or `defaults.context_ratio` |
| `max_output_tokens` | agent override or `defaults.max_output_tokens` |

Use a separate agent when you need a materially different execution profile. That keeps the config understandable and makes delegation explicit.

## Delegation With `spawn_agent`

`spawn_agent(task, context="", agent="")` starts a fresh child run.

Two modes:

- omit `agent` to start a fresh run of the current agent
- set `agent="other-agent"` to switch to that target agent's own prompt, workspace, memory, tools, skills, and permissions

What carries over:

- the current run envelope
- user access restrictions
- the current job or Slack session context

What does not carry over:

- parent conversation history
- parent tool state beyond what the child agent naturally has

Depth is capped at `3` to prevent runaway recursion.

## A Good Default Multi-Agent Pattern

This is the simplest useful split:

- `operator`: Slack-facing, broad coordination, safe tool surface
- `researcher`: headless, strong model, focused research instructions
- `reviewer`: headless, narrow code-review or QA instructions

That gives you clean delegation boundaries without turning one agent into a giant prompt with every responsibility mixed together.
