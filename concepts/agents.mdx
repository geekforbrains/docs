Each agent lives at `~/.operator/agents/<name>/AGENT.md` and gets its own workspace at `~/.operator/agents/<name>/workspace/`.

## Scan First

- `AGENT.md` may include optional YAML frontmatter.
- The markdown body becomes the agent prompt.
- Agents inherit models and execution settings from `defaults` unless overridden.
- `sandbox: true` only affects file tools, not `run_shell`.
- Agents without a transport are headless: usable for jobs and `spawn_agent(...)`, not direct chat.

## Minimal Agent

```markdown
---
name: researcher
description: Investigates technical questions and summarizes findings.
---

You are a research-focused agent.
Prioritize primary sources, concise summaries, and practical next steps.
```

Register the agent in `operator.yaml`:

```yaml
agents:
  researcher:
    models:
      - "openai/o3"
    thinking: high
```

Add a `transport:` block if you want the agent reachable in Slack.

## Frontmatter

| Field | Required | What it does |
| --- | --- | --- |
| `name` | no | Defaults to the directory name |
| `description` | no | Used for agent discovery in other agents' prompts |

If an agent has a `description`, other agents can see it in their injected "Available Agents" list and may choose it when calling `spawn_agent(agent="...")`.

The frontmatter is stripped before prompt injection. The LLM sees only the markdown body.

## Workspace

Each agent runs with:

- working directory: `~/.operator/agents/<name>/workspace/`
- prompt file: `~/.operator/agents/<name>/AGENT.md`
- shared directory symlink: `workspace/shared -> ~/.operator/shared/`

This workspace is used by:

- `run_shell`
- `read_file`
- `write_file`
- `list_files`
- `send_file`
- `run_skill` subprocesses

Files persist across chat turns and job runs.

## Prompt Assembly

Operator assembles the system prompt in a fixed order.

### Stable prefix

1. `~/.operator/SYSTEM.md`
2. the agent prompt body from `AGENT.md`
3. discovered skills (`name` and `description`)
4. discovered peer agents (`name` and `description`)

### Dynamic suffix

1. transport/job context
2. pinned memories
3. transport-specific help, such as Slack messaging guidance and channel lists

Stable and dynamic sections are separated by an internal cache boundary. On Anthropic models, Operator uses that split to apply prompt caching safely.

## Message Timestamp Rendering

Operator stores a nullable `created_at` separately from message content and only stamps `user` messages when building a live model request.

That means:

- chat turns get their inbound message time rendered into the prompt
- scheduled jobs get their run-start time rendered onto the job body
- `spawn_agent(...)` gets the child task time rendered onto the sub-agent's initial `user` message

The rendered format is local to `runtime.timezone`, for example:

```text
[Monday, 2026-03-09T09:22:40-07:00]
```

Assistant, tool, and system messages are not stamped. Older stored messages with `null` `created_at` remain unchanged.

## Sandbox

```yaml
agents:
  operator:
    sandbox: true
```

| Setting | Behavior |
| --- | --- |
| `sandbox: true` | File tools cannot escape the workspace |
| `sandbox: false` | File tools may read/write outside the workspace |

Important caveats:

- `run_shell` is unaffected by sandboxing
- `shared/` is a symlink that resolves outside the workspace, so sandboxed file tools cannot follow it
- shell commands can still access `shared/` because the shell runs from the workspace and follows the symlink normally

Use sandboxing and permissions together for public-facing agents.

## Permissions

```yaml
agents:
  researcher:
    permissions:
      tools: [web_fetch, read_skill, run_skill, spawn_agent]
      skills: [job-creator]
```

Permissions are flat allow-lists.

| Config | Result |
| --- | --- |
| omitted | unrestricted |
| `"*"` | unrestricted |
| list | only listed names are visible |

Tool permissions and skill permissions are independent.

## Sub-Agents

`spawn_agent(task, context="", agent="")` creates a focused child run and returns its final response.

Behavior depends on whether `agent=` is provided:

- no `agent`: inherit the current agent prompt, models, workspace, thinking level, sandbox setting, permissions, and shared dir
- `agent="other"`: switch to the target agent's resolved config

Sub-agents are limited to depth `3`.

The child run's initial `user` message is timestamp-rendered the same way as chat and job requests.

## Headless Agents

Agents without a `transport` block are still useful for:

- scheduled jobs
- `spawn_agent(agent="...")`
- shared model/prompt presets

<Warning>
  The long-running `operator` service still requires at least one transported agent somewhere in config. Headless-only configs cannot start the runtime.
</Warning>

## In Existing Slack Threads

When an agent is first mentioned inside an existing Slack thread, Operator may inject a one-time snapshot of earlier thread replies so the model can see what happened before it joined. That snapshot is request-only and not treated as direct user input.
