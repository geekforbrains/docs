---
title: "Jobs"
description: "Scheduled markdown-defined tasks with cron, hooks, and runtime state"
---

Jobs are stored in `~/.operator/jobs/<name>/JOB.md`.

## Scan First

- Cron schedules are evaluated against the current UTC minute.
- Job prompts are markdown bodies under YAML frontmatter.
- The job body becomes a stamped `user` message when the run starts.
- `prerun` can gate a job by returning non-zero.
- `postrun` receives the final model output on stdin.
- Scheduled jobs only get transport messaging if the target agent has a runtime transport.
- `operator job run` always uses a CLI transport and prints `send_message` output locally.

## Job File Format

```yaml
---
name: daily-summary
description: Summarize yesterday's activity
schedule: "0 9 * * *"
agent: operator
model: "anthropic/claude-sonnet-4-6"
max_iterations: 10
enabled: true
hooks:
  prerun: scripts/check.sh
  postrun: scripts/publish.sh
---

Summarize the key events from the last 24 hours.
Post a short teaser to #general, then reply in a thread with the full summary.
```

The body is the `user` prompt for the job run. Because jobs do not run as an authenticated user, the rendered timestamp on that message uses UTC.

## Scheduler Behavior

The background runtime:

- scans `jobs/*/JOB.md`
- checks cron matches once per minute
- skips disabled jobs
- increments `skip_count` when a previous run of the same job is still active
- serializes jobs per agent, queueing multiple due jobs that target the same agent

## Agent Selection

The execution agent is:

1. `job.agent`, if present
2. otherwise the default agent from config

That chosen agent supplies:

- the prompt body from `AGENT.md`
- permissions
- available skills
- workspace
- `thinking`
- `context_ratio`
- `max_output_tokens`
- the default `models` fallback chain when `job.model` is unset

## What Jobs May Override

Job frontmatter supports two execution overrides:

| Field | Meaning |
| --- | --- |
| `model` | replaces the agent model chain with a single model |
| `max_iterations` | overrides the agent/default iteration cap |

Jobs do **not** support per-job `thinking`, `context_ratio`, or `max_output_tokens`.

## Messaging Behavior

Scheduled jobs use the selected agent's configured transport when one exists.

That means:

- jobs targeting a Slack-connected agent can call `send_message` or `send_file`
- jobs targeting a headless agent still run, but transport tools will error during scheduled execution

`operator job run <name>` is different:

- it does **not** post to Slack
- it injects a CLI transport
- `send_message` output is printed in your terminal

<Note>
  `operator job run` ignores the cron schedule and runs the job immediately, even if the job is disabled.
</Note>

## Hooks

```yaml
hooks:
  prerun: scripts/check.sh
  postrun: scripts/publish.sh
```

| Hook | Behavior |
| --- | --- |
| `prerun` | runs before the LLM call; non-zero exit marks the run as `gated` and skips the model |
| `postrun` | runs after the final model output; receives that output on stdin |

Hook paths must be **relative** and must remain inside the job directory. Absolute paths and `..` escapes are rejected.

Job hooks receive:

| Variable | Value |
| --- | --- |
| `JOB_NAME` | current job name |
| `JOB_DIR` | absolute path to the job directory |
| `OPERATOR_AGENT` | agent running the job |
| `OPERATOR_HOME` | Active Operator base directory for the loaded config |
| `OPERATOR_DB` | path to `operator.db` |

The hook working directory is the job directory. Hook timeout comes from `defaults.hook_timeout` and defaults to `30` seconds.

## Memory In Jobs

Jobs do not use user-scoped memory.

Job recall and rule injection only use:

- `agent/<agent-name>`
- `global`

This keeps scheduled work from depending on a specific human user's private context.

## Runtime State

Each job has persisted runtime counters:

| Field | Meaning |
| --- | --- |
| `last_run` | ISO timestamp of the last attempt |
| `last_result` | `success`, `error`, or `gated` |
| `last_duration_seconds` | rounded duration |
| `last_error` | error string for the last failed run |
| `run_count` | number of runs that reached model execution |
| `error_count` | number of failed runs after execution started |
| `gate_count` | number of prerun-gated runs |
| `skip_count` | number of scheduler skips because a prior run was still active |

## Managing Jobs

- create or edit files manually in `~/.operator/jobs/`
- use the deterministic runtime tools `create_job`, `update_job`, `delete_job`, `enable_job`, `disable_job`, and `list_jobs`
- use the [Jobs CLI](/cli/jobs) to inspect, enable, disable, and manually run jobs
