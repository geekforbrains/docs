Jobs are cron-scheduled agent runs defined in `~/.operator/jobs/<name>/JOB.md`.

Use a job when work should happen on a schedule instead of waiting for a human message.

## The Mental Model

A job is:

- a markdown file with YAML frontmatter
- scanned once per minute
- run as an agent
- tracked with runtime state in SQLite

The job prompt is internal. If you want output posted somewhere, call `send_message` or `send_file` from the job with an explicit destination.

## Example `JOB.md`

```yaml
---
name: daily-summary
description: Summarize the last day of activity and post it to Slack.
schedule: "0 9 * * *"
agent: operator
enabled: true
hooks:
  prerun: scripts/prerun.sh
  postrun: scripts/postrun.sh
---

Summarize the key events from the last 24 hours.
Post the short version to #general and keep the longer breakdown in the thread.
```

Why "explicit destination" matters:

- jobs are not conversations
- there is no current channel or thread to fall back to
- `send_message` and `send_file` only know where to deliver output if you pass `channel=...` yourself

Example:

```python
send_message(
    channel="#general",
    text="Daily summary is ready.",
)
```

Important fields:

| Field | Meaning |
| --- | --- |
| `name` | job name used by the CLI and tools |
| `schedule` | cron expression |
| `description` | human-readable summary |
| `agent` | which agent runs the job; default is the config default agent |
| `model` | optional model override |
| `max_iterations` | optional iteration override |
| `enabled` | whether the job runs on schedule |
| `hooks` | optional `prerun` and `postrun` scripts |

## Create Jobs Through Tools

Agents do not need to compose raw frontmatter by hand. Operator ships typed job tools:

- `create_job`
- `update_job`
- `delete_job`
- `enable_job`
- `disable_job`
- `list_jobs`

Example:

```python
create_job(
    name="daily-summary",
    schedule="0 9 * * *",
    prompt="Summarize the key events from the last 24 hours.\nPost to #general.",
    agent="operator",
    prerun=True,
)
```

When `prerun=True` or `postrun=True`, the tool creates placeholder scripts at:

- `scripts/prerun.sh`
- `scripts/postrun.sh`

You can still edit `JOB.md` directly. The tools exist to keep creation and updates deterministic.

## How Jobs Run

At execution time Operator:

1. resolves the target agent
2. runs the optional `prerun` hook
3. builds a job-specific system prompt
4. injects `<prerun_output>` if the hook produced stdout
5. runs the agent
6. runs the optional `postrun` hook with the final model output on stdin
7. records runtime state

The job-specific system prompt explicitly tells the agent that plain text responses are internal and that it should use `send_message` to deliver output.

It does not supply a default destination. If the job needs to post somewhere, the tool call must include the target channel explicitly.

## `prerun` Hooks

Use a prerun hook for deterministic work:

- data collection
- filtering
- gating
- date math
- API calls that should not be repeated by the model

Behavior:

- stdout is injected into the prompt as `<prerun_output>`
- non-zero exit code gates the run
- gated runs increment the job's gate count

Example:

```bash
#!/bin/bash
set -euo pipefail

curl -fsSL https://example.internal/report.json
```

## `postrun` Hooks

Use a postrun hook when the final output needs a deterministic follow-up step:

- writing to an external API
- validating output
- formatting or packaging artifacts

Behavior:

- receives the agent output on stdin
- non-zero exit marks the run as failed

## Hook Boundaries

Current implementation details:

- hook paths must be relative
- hook paths must stay inside the job directory
- absolute paths are rejected
- timeout comes from `defaults.hook_timeout`

Hook processes receive these environment variables:

- `JOB_NAME`
- `JOB_DIR`
- `OPERATOR_AGENT`
- `OPERATOR_HOME`
- `OPERATOR_DB`

## Runtime State

Operator stores per-job runtime state:

- last run time
- last result
- last duration
- last error
- run count
- error count
- gate count
- skip count

Use the CLI to inspect it:

```sh
operator job list
operator job info daily-summary
```

## Run A Job Manually

```sh
operator job run daily-summary
```

This ignores the cron schedule and runs immediately.

Important detail: manual runs use the CLI transport, so `send_message` output is printed to your terminal instead of being sent to Slack.

That still requires an explicit `channel`. In a manual run, the CLI transport prints the channel label you passed instead of resolving a live Slack destination.

## Good Job Design

Keep the job prompt focused on interpretation and output.

Push deterministic work into hooks when:

- the same API call would happen every run
- the result should gate the run
- the result is better handled in shell than in model reasoning

That keeps jobs faster, cheaper, and easier to debug.
