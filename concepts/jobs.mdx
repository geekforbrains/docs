Jobs are for recurring work that should happen without waiting for a human to ask.

Each job lives at `~/.operator/jobs/<name>/JOB.md`. The file is the source of truth, and the scheduler reads it directly.

## When To Use A Job

Use a job when the work is:

- recurring
- asynchronous
- better expressed as a prompt plus a little deterministic glue

Examples:

- daily summaries
- release reminders
- recurring status reports
- periodic cleanup or sync tasks

## Job File Format

```yaml
---
name: daily-summary
description: Summarize yesterday's activity
schedule: "0 9 * * *"
agent: operator
model: "anthropic/claude-sonnet-4-6"
max_iterations: 10
enabled: true
hooks:
  prerun: scripts/check.sh
  postrun: scripts/publish.sh
---

Summarize the key events from the last 24 hours.
Post a short teaser to #general, then reply in a thread with the full summary.
```

The markdown body becomes the job's `user` message.

Best practice: keep the directory name and frontmatter `name` aligned even though the runtime can still find a mismatched frontmatter name by scanning.

## Scheduler Behavior

The background runtime:

- scans `jobs/*/JOB.md`
- checks cron matches once per minute
- evaluates cron schedules against the current UTC time
- skips disabled jobs
- serializes jobs per target agent through FIFO queues
- allows different agents to run jobs concurrently

Two runtime counters matter:

- `skip_count` increments when the same job is due again while its previous run is still active
- queued jobs for the same agent still run in order once the agent worker is free

## Which Agent Runs The Job

Agent resolution is:

1. `job.agent`, if set
2. otherwise the default agent from config

That agent supplies:

- `AGENT.md`
- permissions
- skill access
- workspace
- memory scope
- `thinking`
- `context_ratio`
- `max_output_tokens`
- default model chain when `job.model` is unset

## What A Job Can Override

Jobs can override only two execution settings:

| Field | Meaning |
| --- | --- |
| `model` | replace the resolved agent model chain with a single model |
| `max_iterations` | override the resolved iteration cap |

Jobs do not support per-job overrides for:

- `thinking`
- `context_ratio`
- `max_output_tokens`

If you need those to differ, point the job at a different agent.

## Hooks

```yaml
hooks:
  prerun: scripts/check.sh
  postrun: scripts/publish.sh
```

Hook behavior:

| Hook | Behavior |
| --- | --- |
| `prerun` | runs before the model; non-zero exit gates the job and skips model execution |
| `postrun` | runs after the final model output; receives that output on stdin |

`prerun` stdout is injected into the prompt as:

```text
<prerun_output>
...
</prerun_output>
```

This is the main reason jobs are practical. Deterministic work stays in scripts. The model works on already-fetched or already-filtered data.

Technical constraints:

- hook paths must be relative
- hook paths must stay inside the job directory
- absolute paths and `..` escapes are rejected
- the working directory is the job directory
- timeout comes from `defaults.hook_timeout`

Environment variables available to hooks:

| Variable | Value |
| --- | --- |
| `JOB_NAME` | current job name |
| `JOB_DIR` | absolute path to the job directory |
| `OPERATOR_AGENT` | agent running the job |
| `OPERATOR_HOME` | active Operator base directory |
| `OPERATOR_DB` | path to the active `operator.db` |

## Transport Behavior

Scheduled jobs use the selected agent's configured transport when one exists.

That means:

- a Slack-connected agent can post with `send_message` or `send_file`
- a headless agent still runs the job, but transport-dependent tools return errors

`operator job run <name>` behaves differently:

- it ignores the cron schedule
- it runs even if the job is disabled
- it uses the CLI transport instead of Slack

So if the job calls `send_message`, you see the output in the terminal instead of in Slack.

## Memory In Jobs

Jobs do not use user-scoped memory.

Available memory scopes during job execution:

- `agent`
- `global`

That keeps scheduled work from depending on one user's private conversation context.

## Runtime State

Each job stores persistent runtime counters in SQLite:

| Field | Meaning |
| --- | --- |
| `last_run` | timestamp of the last attempt |
| `last_result` | `success`, `error`, or `gated` |
| `last_duration_seconds` | rounded duration |
| `last_error` | error string for the last failed run |
| `run_count` | runs that reached execution |
| `error_count` | failed runs |
| `gate_count` | prerun-gated runs |
| `skip_count` | scheduler skips because a previous run of the same job was still active |

Use `operator job list` and `operator job info <name>` to inspect this state quickly.

## Creating And Updating Jobs

You have two supported paths:

1. edit files directly under `~/.operator/jobs/`
2. use the deterministic tools `create_job`, `update_job`, `delete_job`, `enable_job`, `disable_job`, and `list_jobs`

The job tools are safer for agent-authored changes because they take structured fields instead of raw YAML and can scaffold placeholder hook scripts automatically.
