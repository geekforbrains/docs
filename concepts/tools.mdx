title: "Built-in Tools"
description: "The tool surface Operator agents can use at runtime"

Operator registers a built-in tool set and may add transport-specific tools at runtime.

## Scan First

- Permissions can hide tools from specific agents.
- Some tools require runtime context to work correctly.
- Transport tools only appear for the active transport.
- `run_shell` is powerful and unaffected by sandboxing.
- Memory tools need runtime memory to be enabled.

## Tool Availability Rules

There are three separate gates:

1. **Registration:** built-in tools are registered globally
2. **Permissions:** `permissions.tools` may hide them from an agent
3. **Runtime context:** some tools still need a transport, memory store, or agent context

Example: `save_memory` is registered, but it returns `memory system not configured` if runtime memory is off.

## Shell and Files

| Tool | Notes |
| --- | --- |
| `run_shell(command, timeout=120)` | runs `bash -c` from the agent workspace |
| `read_file(path)` | reads a file, capped at 1 MB / 16 KB returned output |
| `write_file(path, content)` | writes a file, creating parent directories |
| `list_files(path=".", max_depth=2)` | lists non-hidden files and directories |

Important behavior:

- file paths resolve relative to the agent workspace
- sandboxed agents cannot escape the workspace with file tools
- unsandboxed agents may use absolute paths or parent traversal
- `run_shell` is not sandboxed, only rooted at the workspace as `cwd`

## Web

| Tool | Notes |
| --- | --- |
| `web_fetch(url, offset=0, limit=16000)` | fetches text-friendly content and paginates large responses |

`web_fetch` prefers:

1. a `.md` variant of the URL when available
2. clean text extraction from HTML
3. raw text for already textual content

It only returns text-like content. Unsupported binary types return an error string.

## Messaging

| Tool | Notes |
| --- | --- |
| `send_message(channel="", text="", thread_id="")` | sends a message and returns the platform message ID |
| `send_file(path, channel="", thread_id="")` | uploads a file and returns the platform message ID when supported |

Defaults:

- if `channel` is omitted, use the current conversation channel
- if `thread_id` is omitted, use the current conversation thread

`send_file` uses the same path-resolution and sandbox rules as file tools and rejects files over 50 MB.

## Sub-Agents

| Tool | Notes |
| --- | --- |
| `spawn_agent(task, context="", agent="")` | creates a focused child run and returns its final answer |

Rules:

- max depth is `3`
- without `agent=`, the child inherits the current run context
- with `agent=`, the child switches to that agent's config

## Memory

| Tool | Notes |
| --- | --- |
| `save_memory(content, scope="user", retention="durable", pinned=false)` | stores a memory |
| `search_memories(query, scope="", top_k=5)` | semantic lookup |
| `forget_memory(memory_id)` | deletes a memory |
| `list_memories(scope="", limit=50, offset=0)` | lists visible memories |

These tools require runtime memory to be active. In public channels, `user` scope is blocked.

## Key-Value Store

| Tool | Notes |
| --- | --- |
| `kv_get(key, namespace="")` | returns a value or `[not found]` |
| `kv_set(key, value, namespace="", ttl_hours=0)` | stores a value with optional TTL |
| `kv_delete(key, namespace="")` | deletes a key |
| `kv_list(namespace="", prefix="")` | lists keys for the current agent |

KV data is always scoped to the current agent.

Use namespaces for grouping operational state like:

- cursors
- watermarks
- job checkpoints
- dedupe keys

## Jobs and Skills

| Tool | Notes |
| --- | --- |
| `manage_job(action, name="", config="")` | list, create, update, delete, enable, disable |
| `manage_skill(action, name="", config="")` | list, create, update, delete |
| `read_skill(skill, path="")` | read `SKILL.md`, references, or assets |
| `run_skill(skill, command, timeout=120)` | execute a skill-related command |

Key behavior:

- `manage_job create/update` expects full `JOB.md` content
- `manage_skill create/update` expects full `SKILL.md` content
- `manage_job` validates cron, agent name, and hook path safety
- `manage_job create` can scaffold placeholder hook scripts
- `run_skill` uses `shell=False`, strips transport secrets, preserves `$OPERATOR_HOME`, and sets `SKILL_DIR`

## User Management

| Tool | Notes |
| --- | --- |
| `manage_users(action, username="", role="", transport="", external_id="")` | runtime user and role management |

`manage_users` is role-gated:

- requires the `admin` role
- stays visible to the LLM
- returns an error if the active user is not an admin

Its `add` action creates the user and assigns a role. Identity linking is a separate `link` action.

## Transport Tools

Transport tools are added at runtime. They do not appear in `operator tools`, which only lists built-in tools.

Slack currently adds:

| Tool | Notes |
| --- | --- |
| `list_channels()` | show cached postable channels |
| `read_channel(channel, count=20)` | fetch recent channel messages |
| `read_thread(channel, thread_id, count=50)` | fetch thread replies |

See [Slack](/transports/slack) for the Slack-specific runtime behavior.

## Choosing The Right Tool

- use **KV** for operational state
- use **memory** for reusable human/project facts
- use **jobs** for recurring autonomous workflows
- use **skills** for reusable instructions and scripts
- use **sub-agents** when a task benefits from isolation or a different agent persona
