Operator registers a built-in tool set and may add transport-specific tools at runtime.

## How Tool Availability Works

Three gates control whether an agent can actually use a tool:

1. the tool must be registered
2. the agent must be allowed to use it through `permissions.tools`
3. the current run must provide any required context such as a transport, memory store, or authenticated user

Example:

- `send_message` is always registered
- it still fails if the current run has no transport

`operator tools` lists only built-in tools. Transport tools such as Slack tools are added at runtime and do not appear in that CLI output.

## Files, Shell, And Web

| Tool | Purpose |
| --- | --- |
| `read_file(path)` | read a file |
| `write_file(path, content)` | write a file |
| `list_files(path=".", max_depth=2)` | list files and directories |
| `run_shell(command, timeout=120)` | run `bash -c` in the agent workspace |
| `web_fetch(url, offset=0, limit=16000)` | fetch URL content as clean text |

Technical details:

- relative file paths resolve from the agent workspace
- absolute paths are allowed
- `read_file` reads up to 1 MB and truncates returned text to 16 KB
- `list_files` skips hidden files and directories
- `web_fetch` prefers `.md` variants when available and notes `llms.txt` support when detected

## Messaging

| Tool | Purpose |
| --- | --- |
| `send_message(channel="", text="", thread_id="")` | post a message and return a platform message ID |
| `send_file(path, channel="", thread_id="")` | upload a file and return a platform message ID |

Defaults:

- omitted `channel` means the current conversation channel
- omitted `thread_id` means the current conversation thread

`send_file` rejects files over 50 MB.

## Memory

| Tool | Purpose |
| --- | --- |
| `save_rule` | create or replace a rule |
| `save_note` | create or replace a note |
| `search_notes` | search notes |
| `list_rules` | list rules |
| `list_notes` | list notes with pagination |
| `read_note` | read one note |
| `forget_rule` | move a rule to trash |
| `forget_note` | move a note to trash |

Memory tools use logical scopes:

- `agent`
- `user`
- `global`

`user` scope is available only in private conversations.

## State

| Tool | Purpose |
| --- | --- |
| `get_state(key)` | read JSON state |
| `set_state(key, value)` | write scalar state |
| `append_state(key, value, max_items=0)` | append to list state |
| `pop_state(key)` | remove and return the oldest list item |
| `list_state()` | list all state keys |
| `delete_state(key)` | delete a state key |

State is file-backed JSON under `agents/<agent>/state/`.

Use state for machine bookkeeping such as:

- cursors
- dedupe keys
- rolling queues
- checkpoints

## Jobs And Skills

| Tool | Purpose |
| --- | --- |
| `create_job`, `update_job`, `delete_job` | manage job definitions |
| `enable_job`, `disable_job`, `list_jobs` | control job state |
| `create_skill`, `update_skill`, `delete_skill`, `list_skills` | manage skills |
| `read_skill` | read skill content or a specific file inside the skill |
| `run_skill` | run a command in skill context |

The job and skill management tools exist so agents do not have to compose raw frontmatter safely by hand.

## Users

| Tool | Purpose |
| --- | --- |
| `manage_users(action, ...)` | list, add, remove, link, unlink, add_role, and remove_role |
| `set_timezone(timezone)` | set the current user's timezone |

Important detail:

- `manage_users` is not separately admin-gated inside the tool implementation
- access control comes from whether the agent itself is allowed to use `manage_users`

Only give it to trusted agents.

## Delegation

| Tool | Purpose |
| --- | --- |
| `spawn_agent(task, context="", agent="")` | run a focused child task in a fresh agent session |

Behavior:

- omitting `agent` spawns a fresh run of the current agent
- specifying `agent` switches to that target agent's own prompt, memory, workspace, tools, skills, and permissions
- current user access restrictions still apply
- max delegation depth is `3`

## Slack Runtime Tools

During Slack-backed runs, Operator adds:

- `slack_find_users`
- `slack_list_channels`
- `slack_read_channel`
- `slack_read_thread`
- `slack_add_reaction`
- `slack_remove_reaction`

These are transport tools, not built-ins. They exist only when the current run is using the Slack transport.

## Choosing The Right Primitive

Use:

- tools when the agent needs to do work now
- skills when the agent needs reusable instructions
- memory when the agent should remember durable knowledge
- state when the runtime needs machine-readable bookkeeping
- jobs when the work should happen on a schedule

Keeping those separate is one of the main reasons Operator stays understandable as a system.
