Every agent has access to these built-in tools:

## Workspace & Files

| Tool | Description |
|------|-------------|
| `run_shell` | Execute shell commands in the agent's workspace |
| `read_file` | Read file contents |
| `write_file` | Write or overwrite a file |
| `list_files` | List files in a directory |

All file paths resolve relative to the agent's workspace at `~/.operator/agents/<name>/workspace/`. By default, paths that escape the workspace are rejected. Set `sandbox: false` on the agent to allow full filesystem access — see [Sandbox](/concepts/agents#sandbox).

## Web

| Tool | Description |
|------|-------------|
| `web_fetch` | Fetch and extract content from a URL |

## Messaging

| Tool | Description |
|------|-------------|
| `send_message` | Send a message to a channel or thread |
| `send_file` | Upload a file from the workspace to a channel or thread |

Both tools default to the current conversation's channel and thread when called without explicit `channel` or `thread_id` parameters. Pass them explicitly to cross-post to a different channel.

## Sub-agents

| Tool | Description |
|------|-------------|
| `spawn_agent` | Spawn another agent as a sub-agent |

## Memory

| Tool | Description |
|------|-------------|
| `save_memory` | Store a memory with scope and optional pin |
| `search_memories` | Semantic search across stored memories |
| `forget_memory` | Delete a specific memory |
| `list_memories` | List memories with filters |

## Key-Value Store

| Tool | Description |
|------|-------------|
| `kv_get` | Retrieve a value by key |
| `kv_set` | Store a value with optional TTL |
| `kv_delete` | Remove a key |
| `kv_list` | List keys and values |

Keys are scoped by agent name and grouped by namespace. Use namespaces to organize related keys (e.g. by job name). Use TTL to auto-expire tracking data.

## Job Management

| Tool | Description |
|------|-------------|
| `manage_job` | Create, update, delete, enable, disable, and list jobs |

## Skill Management

| Tool | Description |
|------|-------------|
| `manage_skill` | Create, update, delete, and list skills |

Actions: `list`, `create`, `update`, `delete`. The `config` parameter takes the full `SKILL.md` content (frontmatter + body). Validates the [agentskills.io](https://agentskills.io) spec on create and update — name format, description length, metadata structure, and non-empty body.

## Skill Access

These tools let agents interact with skills without needing `run_shell` — useful for restricted agents that still need skill capabilities.

| Tool | Description |
|------|-------------|
| `read_skill` | Read skill content (SKILL.md, references, assets) |
| `run_skill` | Execute a command in the context of a skill |

### `read_skill`

Reads files from a skill's directory. Used for progressive disclosure — the agent reads `SKILL.md` first, then reads specific reference or asset files as needed.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `skill` | Yes | Skill name |
| `path` | No | Relative path within the skill directory. Defaults to `SKILL.md`. |

Path traversal is blocked — `..` components are rejected. Only skills in the agent's permission list are accessible.

### `run_skill`

Executes a command in the context of a skill. Runs with `shell=False` (no pipes, redirects, or chaining). Skill script paths are auto-expanded.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `skill` | Yes | Skill name |
| `command` | Yes | Command string (parsed with `shlex`) |
| `timeout` | No | Timeout in seconds (default 120) |

- `SKILL_DIR` env var is set to the skill's absolute path.
- Arguments starting with `scripts/`, `references/`, or `assets/` are auto-prefixed with the skill path.
- Transport tokens and `OPERATOR_*` env vars are stripped from the subprocess environment.
- Output is capped at 16 KB.

## User Management

| Tool | Description |
|------|-------------|
| `manage_users` | Manage users, identities, and roles (admin-only) |

### `manage_users`

Manage users, identities, and roles. Requires the `admin` role — execution is blocked for non-admin users even if the tool is visible.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `action` | Yes | One of: `list`, `add`, `remove`, `link`, `unlink`, `add_role`, `remove_role` |
| `username` | No | Target username |
| `role` | No | Role to add or remove |
| `transport` | No | Transport name (for link/unlink) |
| `external_id` | No | External identity (for link/unlink) |

## Transport Tools

Each transport injects its own tools — only the tools for the active transport are available. For example, an agent connected to Slack gets Slack-specific tools, while a future Discord agent would get Discord-specific tools.

### Slack

| Tool | Description |
|------|-------------|
| `list_channels` | List available channels with IDs and topics |
| `read_channel` | Read recent messages from a channel |
| `read_thread` | Read messages from a specific thread |
