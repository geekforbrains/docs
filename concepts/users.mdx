---
title: "Users & Auth"
description: "How Operator authenticates people, assigns roles, and scopes access"
---

Every inbound message is authenticated against a stored user identity. There is no anonymous mode in the runtime.

## Identity Model

Users are stored as stable usernames:

- lowercase letters, numbers, dots, and hyphens only
- length `1-64`

Transport identities are stored as `transport:external_id`, for example:

```text
slack:U04ABC123
```

The current runtime transport is Slack, but the database format is transport-agnostic.

## Roles

Roles map users to allowed agents.

```yaml
roles:
  team:
    agents: [operator, researcher]
  guest:
    agents: [operator]
```

Rules:

- `admin` is built in and cannot be defined in config
- a user's access is the union of all of their roles
- undefined custom roles can still be stored, but they grant no access until they are defined in config

## Runtime Auth Flow

On each incoming message, Operator:

1. resolves the transport identity such as `slack:U04ABC123`
2. maps it to a username
3. loads that user's roles and timezone
4. resolves the allowed agents from config
5. rejects the message if the target agent is not allowed

If the user has the `admin` role, they can access every configured agent.

## Rejection Behavior

```yaml
runtime:
  reject_response: ignore
```

| Value | Behavior |
| --- | --- |
| `ignore` | silently drop the message |
| `announce` | reply with `You don't have access to this agent.` |

If there are no users in the database, the runtime still starts, but inbound messages are rejected until you add at least one user.

## Timezone

Timezone is stored on the user record, not in `operator.yaml`.

Why that matters:

- live user messages are timestamped in that user's timezone when available
- the timezone appears in prompt context
- jobs do not use user timezone because jobs run without a user context

Users can set their own timezone during chat:

```text
set_timezone("America/Vancouver")
```

## CLI Management

Common commands:

```sh
operator user add gavin --role admin slack U04ABC123
operator user link gavin slack U08ALT111
operator user add-role gavin team
operator user list
operator user info gavin
```

Key distinctions:

- `operator user add` creates the user, assigns a role, and links an identity
- `operator user link` adds another identity to an existing user
- `operator user remove` deletes the user plus linked roles and identities
- `operator user unlink` removes one `transport:external_id` mapping

Current implementation detail:

- `user unlink` requires a positional username for command shape, but the unlink operation removes the exact identity mapping without verifying that it belongs to that username first

## `manage_users`

`manage_users` is the runtime tool version of user management.

Actions:

- `list`
- `add`
- `remove`
- `link`
- `unlink`
- `add_role`
- `remove_role`

Current behavior worth knowing:

- the `add` action creates a user and assigns a role, but does not link a transport identity
- linking is a separate action
- `unlink` removes the exact `transport:external_id` mapping

This is another reason to grant `manage_users` only to trusted agents.

## Memory Scope Interaction

User-scoped memory depends on the authenticated user context.

- private chats can use `user` memory scope
- public channels cannot
- jobs never use `user` scope

That keeps private user memory from leaking into public or scheduled contexts.
