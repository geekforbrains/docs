---
title: "Users & Auth"
description: "How Operator authenticates people and maps them to allowed agents"
---

Every inbound transport message is authenticated. There is no “disable auth” mode in the runtime.

## Scan First

- Unknown users are rejected.
- `admin` is built in and grants access to every agent.
- Custom roles map to explicit agent lists.
- Allowed agents are the union of the user's roles.
- `runtime.reject_response` decides whether rejections are silent or announced.
- User timezone is stored on the user record, not in `operator.yaml`.

## Identity Model

Users are stored as stable usernames:

- lowercase letters, numbers, dots, and hyphens only
- length `1-64`

Transport identities are stored as `transport:external_id`, for example:

```text
slack:U04ABC123
telegram:12345678
```

The current runtime transport is Slack, but the user database can store arbitrary transport prefixes.

## Roles

```yaml
roles:
  team:
    agents: [operator, researcher]
  guest:
    agents: [operator]
```

Rules:

- `admin` is built in and cannot be declared in `operator.yaml`
- multiple roles are allowed
- allowed agents are the union of the listed agent names
- undefined custom roles are stored but grant no access until they are defined in config

`operator user add` warns when you assign a non-admin role that is not currently defined in config.

## Runtime Auth Flow

On each incoming Slack message:

1. resolve `slack:<user_id>` to a username
2. load that user's roles
3. resolve roles to the set of allowed agents
4. reject the message if the target agent is not allowed

If the user has `admin`, access is unrestricted.

## Rejection Behavior

```yaml
runtime:
  reject_response: ignore
```

| Value | Runtime behavior |
| --- | --- |
| `ignore` | silently drop the message |
| `announce` | reply with `You don't have access to this agent.` |

If there are no users in the database at all, the runtime still rejects inbound messages. `operator init` does not create users for you. Bootstrap the first admin explicitly:

```sh
operator user add gavin --role admin slack U04ABC123
```

## Timezone

User timezone is part of the stored user profile.

- it appears in the runtime prompt context
- live `user` message timestamps render in that timezone when present
- jobs do not use it because they run without a user context

Users can store their own timezone in chat with:

```text
set_timezone("America/Vancouver")
```

## CLI Management

```sh
operator user add gavin --role admin slack U04ABC123
operator user link gavin slack U07XYZ999
operator user add-role gavin team
operator user list
operator user info gavin
```

Important distinctions:

- `operator user add` creates the user, assigns a role, and links the identity in one command
- `operator user link` adds another transport identity to an existing user
- `operator user remove` deletes the user, roles, and identities together

See [Users CLI](/cli/users) for full command behavior.

## `manage_users` Tool

The built-in `manage_users` tool is runtime-only and **admin-gated**.

Its `add` action creates a user and assigns a role, but it does **not** link a transport identity. Linking is a separate action:

```text
manage_users(action="add", username="alex", role="team")
manage_users(action="link", username="alex", transport="slack", external_id="U123ABC45")
```

Non-admin users can still see the tool in the model surface, but execution is blocked with an error.

## Memory Scope Interaction

In private conversations, live chat memory uses the authenticated user context for `user`-scoped memory. In public channels, `user` scope is unavailable and only `agent` plus `global` memory is considered.

Scheduled jobs do not use user-scoped memory at all.
