# Local agents, normal files, real work.

Operator runs named agents from `~/.operator/`, connects them to Slack, and keeps the important parts of the system in files you can inspect and edit.

The codebase is opinionated on purpose:

- `AGENT.md` is the agent charter.
- `JOB.md` is the scheduled task definition.
- `SKILL.md` is reusable know-how.
- memory lives in markdown files under `rules/` and `notes/`.
- SQLite stores conversation history, identities, and indexes, but not the human-facing source of truth.

Slack is the only runtime transport in the current code.

## What Lives On Disk

```text
~/.operator/
├── operator.yaml
├── .env
├── SYSTEM.md
├── agents/
│   └── <agent>/
│       ├── AGENT.md
│       ├── workspace/
│       ├── memory/
│       └── state/
├── jobs/
│   └── <job>/
│       └── JOB.md
├── skills/
│   └── <skill>/
│       └── SKILL.md
├── memory/
│   ├── global/
│   └── users/
├── shared/
├── db/
└── logs/
```

Why this matters:

- prompts are versionable
- rules and notes stay readable
- jobs and skills are plain files, not hidden dashboard state
- work artifacts live in the workspace instead of disappearing into model history

## Choose The Right Primitive

| Use this | When you want |
| --- | --- |
| `AGENT.md` | a different role, mission, or standing behavior |
| `permissions` | a tighter or broader execution boundary |
| memory `rules/` | short instructions that should shape every future run |
| memory `notes/` | durable knowledge that should be searched on demand |
| state | machine bookkeeping like cursors, queues, and checkpoints |
| skills | reusable instructions, references, scripts, and assets |
| jobs | recurring work on a cron schedule |

## Why The File-First Model Works

- It is reviewable. You can diff prompt, job, and skill changes in git.
- It is debuggable. If a rule exists, you can open the file that created it.
- It is recoverable. The important state is still understandable without a web UI.
- It keeps memory honest. Durable knowledge stays visible and scoped.
