# Agents have joined the chat.

Operator is a local runtime for autonomous agents that live in Slack, use tools, remember context, and run scheduled jobs. Slack is the only transport today. Agents, jobs, skills, and memory stay as plain files under the active Operator home, usually `~/.operator/`.

## Scan First

- Slack-native today: DMs and `@mentions` in invited channels, with thread-scoped sessions.
- Local-first: state lives under the active Operator home, usually `~/.operator/`, with SQLite in `db/operator.db`.
- File-driven: agents use `AGENT.md`, jobs use `JOB.md`, skills use `SKILL.md`, and memory uses markdown under `rules/` and `notes/`.
- Team-aware: every inbound transport message is authenticated against users and roles.
- Model-agnostic: model names use LiteLLM format like `anthropic/claude-sonnet-4-6`.
- Tooling is real: shell, files, web, messaging, jobs, skills, memory, state, and sub-agents.

## What You Get

- **Agents as teammates.** Each agent has its own prompt, workspace, model settings, permissions, and optional Slack transport.
- **Fresh run-mode-preserving delegation.** Agents can call `spawn_agent(...)` to break work into focused child runs, including cross-agent delegation, without copying parent conversation history.
- **Jobs.** Scheduled markdown-defined jobs run every minute against UTC cron schedules.
- **Users and roles.** Unknown users are rejected. Allowed agents are the union of the user's assigned roles. `admin` is built in.
- **File-backed memory.** Rules are always injected, notes are searched on demand, and expired entries move to `trash/`.
- **Per-agent state.** Agents can keep operational state in JSON files under `agents/<name>/state/`.
- **Fallback chains and simple reasoning levels.** Configure `models:` as a failover list that also covers unusable model responses, and `thinking:` as `off|low|medium|high`.

## How A Request Flows

1. A Slack DM, Slack `@mention`, or scheduled job starts a run.
2. Operator loads config, resolves the agent, and assembles the system prompt from `SYSTEM.md`, the agent's `AGENT.md` body, discovered skills, configured peer agents, transport context, and rule memory.
3. It renders creation timestamps onto live `user` messages using the authenticated user's timezone when available, otherwise UTC.
4. The agent runs a model/tool loop until it returns a final answer or hits `max_iterations`.
5. Conversations, identities, and job state are stored in SQLite. File-backed memory and agent state remain on disk.

Slack chat runs are intentionally thread-scoped: a top-level DM or top-level channel mention starts a fresh session thread, and later channel replies only count as new input when they mention the agent again.
