# Agents have joined the chat.

Operator is a local runtime for autonomous agents that live in Slack, use tools, remember context, and run scheduled jobs. The current transport is Slack. Agents, jobs, and skills are plain markdown files under `~/.operator/`.

## Scan First

- Slack-native today: DMs and `@mentions` in invited channels.
- Local-first: state lives in `~/.operator/`, with SQLite in `~/.operator/state/operator.db`.
- File-driven: agents use `AGENT.md`, jobs use `JOB.md`, skills use `SKILL.md`.
- Team-aware: every inbound transport message is authenticated against users and roles.
- Model-agnostic: model names use LiteLLM format like `anthropic/claude-sonnet-4-6`.
- Transport/runtime tools are real: shell, files, web, messaging, jobs, skills, memory, KV, and sub-agents.

## What You Get

- **Agents as teammates.** Each agent has its own prompt, workspace, model settings, permissions, and optional Slack transport.
- **Sub-agent delegation.** Agents can call `spawn_agent(...)` to break work into focused sub-tasks, including cross-agent delegation.
- **Jobs.** Scheduled markdown-defined jobs run every minute against cron schedules in your configured timezone.
- **Users and roles.** Unknown users are rejected. Allowed agents are the union of the user's assigned roles. `admin` is built in.
- **Optional long-term memory.** When memory services are enabled, Operator injects pinned and semantically relevant memories into live requests.
- **Fallback chains and simple reasoning levels.** Configure `models:` as a failover list and `thinking:` as `off|low|medium|high`.

## File Layout

```text
~/.operator/
├── operator.yaml              # Main config
├── .env                       # API keys and transport secrets
├── SYSTEM.md                  # Global system preamble
├── shared/                    # Shared files for all agents
├── logs/
│   └── operator.log
├── state/
│   └── operator.db            # SQLite state
├── agents/
│   └── <agent>/
│       ├── AGENT.md
│       └── workspace/
├── jobs/
│   └── <job>/
│       ├── JOB.md
│       └── scripts/
└── skills/
    └── <skill>/
        ├── SKILL.md
        ├── scripts/
        ├── references/
        └── assets/
```

## How A Request Flows

1. A Slack DM, Slack `@mention`, or scheduled job starts a run.
2. Operator loads config, resolves the agent, and assembles the system prompt from `SYSTEM.md`, `AGENT.md`, skills, and agent metadata.
3. It renders creation-time stamps onto `user` messages in the configured timezone, for example `[Monday, 2026-03-09T09:22:40-07:00]`, and injects any conversation context it needs for the run.
4. The agent runs a model/tool loop until it returns a final answer or hits `max_iterations`.
5. Messages, job state, KV data, and memory state are stored in SQLite.
